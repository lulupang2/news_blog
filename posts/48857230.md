# 에맥스에서 모든 걸 서비스처럼 만들어버리기

에맥스에 regard하는 사람은 많겠지만, 에맥스가 운영체제 같은 건 사실임. 그냥 운영체제와 비교해서 설명해도 되긴 하지만, 에맥스가 운영체제 커널 위에서 애플리케이션과 유틸리티들을 오케스트레이션하는 능력때문에 그렇게 비교하는 것 같음.

에맥스에서 서비스가 어떻게 동작하는지 보여주는 그림이 있고, 그림을 보면 에맥스가 운영체제와 어떻게 관련되어 있는지 보이는데, 에맥스는 운영체제의 시스템 서비스들(파일 시스템, 네트워크 등)에 접근할 수 있고, 다른 프로그램들을 실행시킬 수도 있음. 이러니까 에맥스 사용자들이 에맥스 안에서 다양한 작업을 거의 모두 처리할 수 있는 거임.

에맥스의 이런 성질때문에, 에맥스 사용자들은 대부분의 컴퓨팅 작업을 에맥스 안에서만 해결함. 이런 점이 '에맥스 안에서만 살아가는 것' 같은 느낌을 준다고 하셈.

이 포스트에서는 에맥스가 어떻게 클라이언트를 구축하고, 서비스를 이용할 수 있는지 몇 가지 방법을 다뤄볼게.

## 클라이언트-서버 모델

클라이언트-서버 모델은 컴퓨터 상호작용 패턴 중 하나로, 리소스를 제공하는 서비스와 리소스를 요청하는 클라이언트 사이에서 작업이 분할됨. 클라이언트가 서버에 요청을 보내면, 서버는 응답을 보내주는 방식임.

이런 클라이언트-서버 모델은 네트워크를 이용하거나 로컬 시스템에서 수행할 수 있음. 네트워크를 이용하는 경우가 가장 대표적이는데, REST 스타일의 소프트웨어 아키텍처를 사용하는 경우가 있음.

에맥스에서 클라이언트를 만들 수 있게 해주는 라이브러리가 여러 개 있음. 대표적으로 UI 관련해서는 미니버퍼, 버퍼, 완성, 탭ulated 리스트 모드, 변수 피치 테이블, 트랜지언트 등이 있고, 클라이언트 에지가 되는 네트워크 관련해서는 URL, 소켓, SMTP, 직렬화/역직렬화 등이 있음. 로컬 데이터베이스 관련해서는 컬렉션, 어소시에이션 리스트, 프로퍼티 리스트, 해시 테이블, SQLite 등이 있음.

클라이언트를 구축하는 데 필요한 라이브러리는 에맥스 내장 라이브러리와 서드파티 라이브러리가 있음. 내장 라이브러리로는 UI, 클라이언트 에지, 로컬 데이터베이스 관련 라이브러리가 있음.

## 에맥스 내장 라이브러리

에맥스에서 사용할 수 있는 내장 라이브러리는 다음과 같음:

*   UI: 미니버퍼, 버퍼, 완성, 탭ulated 리스트 모드, 변수 피치 테이블, 트랜지언트
*   클라이언트 에지: URL, 소켓, SMTP, 직렬화/역직렬화
*   로컬 데이터베이스: 컬렉션, 어소시에이션 리스트, 프로퍼티 리스트, 해시 테이블, SQLite

에맥스 사용자들은 이런 라이브러리들을 이용해서 거의 모든 컴퓨팅 작업을 에맥스 안에서 처리할 수 있음. 이러니 에맥스를 운영체제라고 불러도 되지 않을까? 하는 말이 있음.

## wttr.in 클라이언트 예시

wttr.in은 콘솔 기반의 날씨 서비스임. 에맥스에서 wttr.in을 이용하는 예시를 보셈. 에맥스에서 wttr.in을 이용해서 현재 날씨를 알아보는 코드가 있음. 이 코드는 에맥스에서 미니버퍼를 이용해서 위치를 입력받고, 그 위치의 날씨를 보여주는 예시임.

에맥스에서 wttr.in을 호출하는 함수를 만들 수 있고, 이 함수는 원하는 위치의 날씨를 보여줌. 이러면 에맥스 사용자들은 에맥스 안에서만 거의 모든 작업을 처리할 수 잇음.
