## 야, C/C++도 이제 '안전'할 수 있다? Fil-C가 뭐길래?

어이 젊은 친구들, 요즘 Fil-C라고 메모리 안전한 C/C++ 만든다는 놈들이 난리도 아니지? 원본은 좀 복잡하니까, 할애비가 아주 쉽게 풀어서 설명해줄게. 잘 들어봐. 

기존 C/C++은 포인터 하나 던져주면 그게 낭떠러지인지 지옥인지 모르고 그냥 가잖아? Fil-C는 이 포인터 옆에다가 **'계급장(AllocationRecord)'**을 하나씩 붙여주는 게 핵심이야.

---

### 1. 포인터 뒤에 따라붙는 '그림자 변수'

쉽게 말해서 네가 `void f() { T1* p1; }`이라고 코드를 짜면, Fil-C 컴파일러가 이걸 아래처럼 바꿔버려.

```c
void f() {
    T1* p1; 
    AllocationRecord* p1ar = NULL; // 이놈이 바로 계급장이다
}
```

이 `AllocationRecord` 안에는 이 메모리가 어디서부터 어디까지인지(`visible_bytes`), 그리고 그 안에 포인터가 들어있으면 그놈들 계급장은 어디 있는지(`invisible_bytes`) 같은 정보가 꽉꽉 들어차 있어. 포인터 복사할 때? 당연히 계급장도 같이 복사하지. `p1 = p2;` 하면 `p1 = p2, p1ar = p2ar;` 이렇게 되는 거야. 꼼짝마라지.

### 2. 힙(Heap) 메모리? '보이지 않는 바이트'로 잡는다

로컬 변수는 컴파일러가 관리하면 되는데, `malloc`으로 만든 힙 메모리는 어떡할까? 여기서 Fil-C의 진가가 나오는데, 메모리 할당할 때 요청한 크기 말고도 **'메타데이터 저장용 공간'**을 따로 더 만들어.

*   `visible_bytes`: 네가 쓸 진짜 데이터
*   `invisible_bytes`: 그 데이터 안에 숨은 포인터들의 '계급장'을 저장하는 배열

만약 네가 `p1 = *p2;` 이렇게 포인터를 읽어오려고 하면, Fil-C는 `p2ar->invisible_bytes`를 뒤져서 `p1`의 계급장(`p1ar`)을 찾아와. **포인터가 가는 곳엔 언제나 계급장이 따라간다** 이 말이야.

### 3. 충격과 공포: C/C++에 등장한 Garbage Collector (GC)

이게 제일 골 때리는 부분인데, Fil-C는 **GC(쓰레기 수집기)**를 돌려. 

원래 C는 `free` 안 하면 메모리 줄줄 새잖아? Fil-C는 `AllocationRecord`를 추적하다가 아무도 안 쓰는 놈이 있으면 지가 알아서 `free` 해버려. 심지어 로컬 변수 주소를 따서 밖으로 돌리면(Dangling pointer 위험), 아예 그 변수를 스택이 아니라 힙에다 박아버리고 나중에 GC가 치우게 만든다니까? 이거 완전 자바나 고(Go) 아니냐? 껄껄.

### 4. 그래서 이거 실전에서 쓸 수 있냐?

당연히 세상에 공짜는 없지. 성능은 좀 구려질 거야. 하지만 이런 상황에선 개꿀이지:

*   **내 코드가 언제 터질지 모르는 시한폭탄일 때:** 일단 Fil-C로 돌려서 메모리 버그 싹 잡고, 나중에 Rust나 Go로 갈아타기 전 임시방편으로 딱이야.
*   **컴파일 타임 안전성:** Zig 같은 언어에서 컴파일 할 때 안전하게 코드 돌려보고 싶을 때 쓸만하지.
*   **포인터 출처(Provenance) 확인:** "이 포인터가 진짜 거기서 온 놈 맞아?" 하고 따질 때 아주 명확한 증거가 되지.
