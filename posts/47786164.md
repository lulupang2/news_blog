## 1. 개요
안녕 친구들아. 개발하다 보면 터미널에서 로그 보고 텍스트 굴리는 게 일상이지? 내가 이번에 터미널에서 텍스트 좀 찰지게 보려고 페이저(Pager)를 하나 직접 깎아왔어. 이름은 **lore**라고 지었지.

내가 예전에 k8s 로그 보는 `kl`이나 Nomad용 `wander` 같은 TUI(Terminal UI) 툴들을 좀 만들었거든? 그러다 보니 텍스트 덩어리를 요리조리 내비게이션 하는 기능이 맨날 필요하더라고. 그래서 이번에 아예 Go 언어로 재사용 가능한 뷰포트(viewport) 컴포넌트를 제대로 만들고, 그걸로 내 전용 페이저까지 만들어버렸어. 이제 난 `less` 안 쓰고 내가 만든 거 쓴다.

## 2. 터미널 페이징이 대체 뭔데?
니들 터미널에서 `cat file.txt` 하면 텍스트가 화면 끝까지 쭈르륵 올라가버리잖아? 근데 내용이 수만 줄이면 어떡할 거야? 그때 필요한 게 바로 **페이저(Pager)**야.

*   **터미널의 본질**: 터미널은 기본적으로 격자(Grid)야. 행(Row)과 열(Column)로 이루어진 고정폭 폰트의 세계지.
*   **ANSI 색상 코드**: 터미널에서 색깔 나오는 건 `\x1b[31m` 같은 이스케이프 코드 덕분이야. 이거 파싱하는 게 은근 귀찮아.
*   **$PAGER 환경변수**: `git diff`나 `man` 페이지 볼 때 시스템이 `$PAGER`에 등록된 프로그램을 실행해 주는 거야. 보통은 `less`가 기본이고, 좀 친다 하는 애들은 `bat`이나 `delta` 쓰기도 하지.

## 3. TUI (터미널 UI) 부심 좀 부려보자면
TUI는 웹사이트나 데스크톱 앱이랑은 감성이 달라. 픽셀 단위가 아니라 **터미널 셀** 단위로 돌아가거든. 이게 제약이 빡세긴 한데, 오히려 본질적인 정보만 딱 보여주기엔 최고야. 키보드만으로 모든 걸 제어하는 그 쾌감... 알지?

내가 만든 k8s 로그 뷰어 `kl`을 예로 들면, 왼쪽엔 리소스 트리가 있고 오른쪽엔 로그가 나오는데, 이게 다 내가 만든 뷰포트 컴포넌트를 응용한 거야. 검색 기능(`/`), 매칭되는 것만 보기(`x`), JSON 예쁘게 보기(`p`) 같은 기능을 다 때려 박았지.

## 4. 유니코드... 이 새끼는 진짜 '지옥'이다
개발하면서 제일 빡쳤던 게 뭔지 알아? 바로 **유니코드 처리**야. 이게 왜 문제냐면:

1.  **너비가 제각각임**: 영어는 1칸인데, ✨(이모지)나 '全'(한자)은 2칸을 먹어.
2.  **결합 문자**: `é` 같은 글자는 코드 포인트 하나일 때도 있고, `e`랑 `´`가 합쳐진 경우도 있어. 
3.  **계산 오류**: 이거 계산 잘못하면 터미널 화면 다 깨지고 난리 난다. 

내가 이거 해결하려고 Go에서 문자열 바이트랑 실제 터미널 셀 너비를 일일이 매핑하는 로직 짜느라 머리 다 빠지는 줄 알았다. `wcwidth` 라이브러리 써서 너비 계산하고, 가변 너비 폰트 지옥에서 살아남았지.

## 5. 나도 한번 써볼 수 있냐?
당연하지. Go 설치돼 있으면 이거 한 줄이면 바로 실행해 볼 수 있어.

```bash
go run github.com/robinovitch61/viewport/examples/filterableviewport@latest
```

도커로도 돌아가니까 함 봐바. 검색 기능이랑 텍스트 래핑 기능 같은 거 써보면 꽤 쓸만할 거야.

## 6. 결론
세상엔 `less` 같은 훌륭한 툴이 이미 많아. 하지만 가끔은 이렇게 밑바닥부터 부품(컴포넌트)을 직접 깎아보는 게 공부도 많이 되고 재밌어. 니들도 맨날 남이 만든 것만 가져다 쓰지 말고, 너만의 툴을 한번 만들어봐라. 늙은이 말 들어서 손해 볼 거 없다. 질문 있으면 댓글 달아라!
