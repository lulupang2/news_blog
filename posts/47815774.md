## 1. 개요: 클라우드 쓰다가 파산하게 생겼다

야, 너네 아직도 디지털오션(DO)에 생돈 갖다 바치고 있냐? 터키에서 소프트웨어 회사 운영하는 형님들이 리라화 가치 폭락하고 물가 치솟으니까 인프라 비용 때문에 눈물이 난다더라. 

기존에 DO에서 월 **$1,432(약 190만 원)** 내고 있었는데, 이번에 독일의 갓성비 서버 **Hetzner(헤츠너)** AX162-R로 옮기면서 **월 $233(약 31만 원)**으로 줄였대. 일 년이면 2천만 원 가까이 아끼는 거야. 이거 안 옮기고 배기겠냐?

### 사양 비교 (실화냐?)
| 항목 | DigitalOcean | Hetzner AX162-R |
| :--- | :--- | :--- |
| CPU | 32 vCPU | **AMD EPYC 9454P (48C/96T)** |
| RAM | 192 GB | **256 GB DDR5** |
| SSD | 600GB + 2TB Volume | **1.92 TB NVMe Gen4 RAID1** |
| **월 비용** | **$1,432** | **$233** |

성능은 압도적인데 가격은 1/6 토막이야. 형이 말하는데, 인프라 생태계 기능 안 쓰고 순수하게 서버만 돌리는 '스테디 스테이트' 워크로드면 당장 전용 서버(Dedicated Server) 알아봐라.

## 2. 우리가 돌리던 빡센 스택

이게 무슨 장난감 프로젝트가 아니야. 꽤 무거운 서비스였다고.
*   **MySQL:** 30개 DB (248 GB 데이터)
*   **Nginx:** 사이트 34개
*   **기타:** GitLab EE, Neo4j, Supervisor 백그라운드 워커 수십 개
*   **OS:** CentOS 7 (시체) → AlmaLinux 9.7 (최신)

---

## 3. 서비스 중단 없는 6단계 이주 전략

사용자들 모르게 조용히, '제로 다운타임'으로 옮기는 게 핵심이야. 무지성으로 DNS 바꾸고 기도하는 건 하수들이나 하는 짓이지.

### Phase 1: 새 서버 세팅
Nginx 소스 컴파일하고 PHP 설정 복사하고 똑같이 맞춰놔. SSL 인증서는 그냥 `/etc/letsencrypt/` 통째로 `rsync` 떠서 옮긴 다음, 나중에 `certbot renew --force-renewal` 한 방 때리면 끝이야.

### Phase 2: 웹 파일 복사
`rsync` 써서 65GB(파일 150만 개)를 옮겨. 마지막 컷오버 직전에 증분 동기화(Incremental sync) 한 번 더 돌려서 빠진 거 없나 체크하는 건 기본이지?

### Phase 3: MySQL 마스터-슬레이브 복제
제일 중요한 부분이야. DB 끄고 덤프 뜨면 서비스 죽잖아? 그래서 기존 서버를 마스터, 새 서버를 읽기 전용 슬레이브로 설정해서 실시간 동기화를 걸었어. `mydumper` 썼는데 이거 ㄹㅇ 물건이니까 메모해라.

### Phase 4: DNS TTL 조절
이주 1시간 전부터 TTL(Time To Live)을 3600초에서 300초로 낮춰놔. 그래야 IP 바꿨을 때 전 세계에 빨리 퍼진다.

### Phase 5: Nginx 리버스 프록시 신공
DNS가 퍼지는 동안에도 어떤 애들은 구 서버 IP로 접속할 거 아냐? 그래서 구 서버 Nginx 설정을 파이썬 스크립트로 싹 돌려서 새 서버 IP로 쏴주는 프록시로 바꿔버렸어. 유저는 자기가 어디로 접속하는지도 모르게 되는 거지.

### Phase 6: 최종 컷오버
DNS IP 바꾸고 일주일 지켜보다가 문제없으면 구 서버 끄면 끝!

## 4. MySQL 이주할 때 개고생한 썰 (꿀팁)

### mydumper 써라, 두 번 써라
일반적인 `mysqldump` 쓰면 며칠 걸릴 거, `mydumper`로 32개 스레드 돌리니까 몇 시간 만에 끝남. 병렬 처리가 최고다.

### MySQL 5.7 → 8.0 업그레이드 트러블
CentOS 7 쓰느라 옛날 버전 썼는데, 8.0으로 올리니까 `sys` 스키마가 꼬이더라고. `DROP DATABASE sys;` 하고 강제 업그레이드(`--upgrade=FORCE`) 돌려서 해결했어.

### 'SUPER' 권한의 함정
슬레이브 서버에 `read_only = 1` 걸어놨는데 자꾸 쓰기가 되는 거야. 알고 보니까 앱 유저들한테 `SUPER` 권한이 있더라고? MySQL에서 `SUPER` 권한은 읽기 전용 설정을 씹어버린다. 당장 `REVOKE` 해서 막아버렸지.
