# 포스트그레스ql과 OOM 키勒
     포스트그레스ql은 리눅스에서 돌아가고, 리눅스는 프로세스에게 실제 메모리보다 더 많은 가상 메모리를 할당할 수 있게 해줌.
     그러나 이건 장점이자 약점인 것 같음.
     왜냐하면 프로세스가 물리적 메모리를 실제로 접근하기 전까지는, 커널이 메모리를 백업하지 않으니까.
     메모리가 부족해지면, OOM 키勒가 발동해서 메모리를 확보하려고 프로세스를 종료시킴.
     대부분의 프로세스에서는, OOM 키勒를 처리하는 건 문제가 안되지만, 포스트그레스ql은 달라.
     포스트그레스ql의 백엔드 프로세스는 메모리를 공유하기 때문에, OOM 키勒가 발동하면 데이터가 손상될 수 있음.
     
     # OOM 키勒와 포스트그레스ql의 문제점
     포스트그레스ql의 백엔드 프로세스는 공유 메모리 세그먼트를 가지고 있는데, 이 세그먼트는 WAL 버퍼, 락 테이블 등이 들어가 있음.
     OOM 키勒가 발동하면, 이 세그먼트가 손상되어 데이터가 훼손될 수 있음.
     때문에 포스트그레스ql은 OOM 키勒를 절대로 피할 수 있어야 함.
     
     #.Strict Overcommit
     리눅스에는 3 種類의 메모리 오버커밋 정책이 있음.
     MODE 0(하이브리드)은 기본값으로, MODE 1은 언제나 오버커밋을 허용하고, MODE 2는 스트릭트 오버커밋을 허용하지 않음.
     포스트그레스ql은 MODE 2를 쓸 것을 권장함.
     왜냐하면, OOM 키勒를 피할 수 있고, 데이터가 손상되는 것을 막을 수 있으니까.
     
     # 커널 버그와 648 GB의 팬텀 메모리
    Strict 오버커밋을 설정할 때, 커널 버그 때문에 648 GB의 팬텀 메모리가 발생할 수도 있음.
     이걸 피하려면, 커널 버그를 수정해야 함.
     
     # 결론
     포스트그레스ql에서 OOM 키勒가 발동되는 것을 피하는 가장 좋은 방법은, 스트릭트 오버커밋을 설정하는 것임.
     OOM 키勒를 피할 수 있을 뿐만 아니라, 데이터가 손상되는 것을 막을 수 있으니까.
