### Jujutsu로 Git Rigour Fatigue 극복하기

Git으로 대형 기능 개발할 때, 좋은 커밋 메시지를 작성하기가 힘듦. 
좋은 커밋 메시지란, 예를 들어 tipo 정의 추가, DB 함수 추가, 서버 CRUD, 클라이언트 API, 클라이언트 UI 등으로 나누는 거야. 
그럼 리뷰어가 작은 단위로 풀 리퀘스트를 체크할 수 있어.

근데 실제로는
tipo 정의 추가, DB 함수 추가, 일시적 테스트 코드, 서버 CRUD, 클라이언트 API와 UI 추가, DB 함수 修复, UI 버그 修复, CRUD 수정, 또 다른 UI 버그 修复
이러고 Latter 커밋이 앞 커밋을 덮어씀. 
스토리가 깨지니까.

Jujutsu가 커밋 관리에 도움이 되긴 하지만, 그래도 노력해야 하고, 나는 귀찮아지거든.

jj absorb나 jj squash -i가 약간 도와주긴 하지만, 한계가 있음.
absorb는 가장 최근에 해당 파일을 수정한 커밋에 할당되고, 항상 해당 변경 사항이 있어야 할 커밋과 일치하지는 않음. 
squash는 경계가 cực도로 깨끗하지 않으면 머지 충돌 지옥에 빠지게 됨.

だから, Git Rigour Fatigue의 문제를 해결하는 방법을 찾아냈어. 
예를 들어 커밋을 이렇게 표현해 볼게:
```
mayhem. 첫 번째 커밋은 tipo와 UI가 뒤 섞였어.
```
이 문제를 해결하려면, 우리가 원하는 커밋 기록을 jj new -B로 생성하고,
 JJ new -A로 남은 커밋들을 처리해.
jj squash --from ~ --into로 실제 변경 사항이 있는 커밋들을 한 곳으로 합친다음,
jj squash -i로:red changes들을 red 박스로 옮겨. 
이ように 해서, 모든 게 제 자리에 있고, everything 커밋은 텅 비게 됨.

대형 기능 개발할 때, 이 워크플로우가 Git Rigour를 엄격하게 지키는 것보다 훨씬 편해.
임시 디버깅 상태가 포함된 커밋을 만들고, 그 모든 걸 한 번에 정리하기가 훨씬 쉽거든.

이 기법에 좋은 이름이 없，不过 '큰 세탁물처럼 커밋하기' 어떠세요?
jj split과는 다르다고 생각해. 
split을 사용하면, 내가 빨간색 커밋에 들어가야 하는 파일을 빼먹으면 다시 split하고 squash해야 해.
이 기법은 가장 쉬운 파일부터 분류하기 시작해도, 커밋 순서에 영향을 받지 않아.

jj squash -i를 사용하면서도Conflict가 발생하지 않는다는 장점이 있음. 
마지막 커밋이 Conflict가 발생하지 않는 게 보장되니까.
새로운 '빨간색과 초록색 수정' 커밋을 만들고, 그 커밋을 빨간색과 초록색 커밋에 interactively squash하면, blue 커밋이 영향을 받을 수 있어.

단점은, 모든 커밋이 컴파일 되지 않을 수도 있다는 점이야. 
그렇지만, 대형 기능 개발할 때, 이 워크플로우가 최고인것 같아.
