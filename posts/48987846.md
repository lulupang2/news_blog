### 소개
도둔은 93년에 나온 게임으로 id 소프트웨어가 만들었음. 이건 게임계에 혁명을 불러일으켰고 现代 FPS를 정의하는 데 기여했음. 이렇게 유명해지게 된 이유는 '도둔은 아무거나 달고 다닐 수 있다'라는 말에서 비롯된 듯. 이 말이 맞을까 보려고 도둔을 거의 모든 기기에서 실행해 봤음. 마이크로컨트롤러, 토스터기, 심지어 박테리아에서도 돌린 사람이 있대 ㅋㅋ. 두 주 전에는我們가 만든 프로세서에서 도둔을 실행하는 영상 올렸었음. 그게 몇백만 뷰가 넘게 되는 기염을 토했음. 진짜 내가 자랑스러울 때가 이런 때임 ㅋㅋ.

### 시작
우리의 목표는 더 복잡한 프로그램을 실행하는 거였음. 팡 게임은 재밌었지만 70년대 게임이라 더现代적인 게임을 원했음. 하지만 두 가지 거대한 장벽이 있었음. 하나는 메모리였음. 큰 프로그램은 더 많은 메모리가 필요했지만 우리가 쓴 FPGA의 BRAM은 메가바이트도 못 되어서 도둔-shareware 버전인 doom1.wad이 14메가바이트인데, 이건 프로그램 실행에 필요한 메모리는 포함하지도 않음. 두 번째 장벽은 속도였음. 도둔은 현대 컴퓨터에서는 문제 없지만 우리의 프로세서에서는 속도가 너무慢임.

그런데 내가 리암과 함께 이렇게 어려운 작업을 시작하게 된 것은 우리가 프로세서를 만들 때부터 이런 문제를 해결하려고 했기 때문임. 리암은 지금 출순 처리하는 중이고 이게 더 많은 병렬처리와 파이프라인 Tricks를 가능하게 해 줌. 메모리 통합은 내가 맡았고, 이렇게 설명하면 간단하지만 실제로 구현하기에는 상당히 복잡함.

### 메모리 통합
원래 우리가 만든 프로세서는 메모리가 매우 간단했음. FPGA의 BRAM은 1サイクル 지연시간이 있고 매우 간단하게 인터페이스할 수 있음. 이 일관성은 우리의 파이프ไลน 프로세서가 메모리를 위해 절대 기다리지 않았음. 또한 BRAM은 단어 단위로 읽고 수정할 수 있었음.

반면에 DDR3 메모리는 느렸고 가변 지연시간이 있었고 넓은 버스 폭이 있었음. 이런 점들이 메모리 작동을 복잡하게 만들었음. 캐시를 사용해서 프로그램이 모든 메모리를同时 사용하지는 않으니까, 활성 메모리 영역을 BRAM에 저장해서 접근 속도를 증가시킬 수 있었음. 잘 최적화된 캐시는 추가적인 지연을 거의 없앨 수 있었음.

### 디자인
이 버전의 CPU는 비교적 표준적인 5단계 파이프라인을 사용했음: 명령어 가져오기, 디코딩, 레지스터 읽기, 실행, 쓰기. 또한 메모리 작동을 단일 인터페이스로 추상화해서 핵심 파이프라인 단계를 단순화했음.

### 핵심 파이프라인
명령어 가져오기 단계는 프로그램 포인터를 추적하고 메모리에서 올바른 명령어를 가져옴. 이전 디자인과 다르게 내부적으로 리ダイ렉션과 정지를 처리했음. 하지만 새로운 디자인에서는 메모리 요청이 이미 보내졌을 때 리ダイ렉션 요청을 받으면, 메모리 응답이 올 때까지 기다려야 했음. 결국 이렇게 고쳐서 명령어 가져오기 단계를 잘 작동 시켰음.

디코딩 단계는 간단했음. 32비트 명령어를 분해해서 필요한 부분을 저장했음.

읽기 단계는 상당히 수정했음. 이전 레지스터 파일은읽기 작업이 너무 느렸음. 그래서 이번에는 읽기 작업을 파이프라인화했음. 레지스터 사용 맵을 사용해서 읽기 작업을 quyết정했음.

실행 단계는 명령어를 처리하는 곳이었음. 여기서 분기나 메모리 접근 같은 작업을 처리했음. 메모리 접근할 때는 따로 메모리 대기 단계를 만들어서 메모리 응답을 기다렸음.

쓰기 단계는 간단했음. 값을 레지스터 파일에 다시 저장하는 것이었음.

### 캐시
캐시는 매우 간단했음. I캐시와 D캐시는 사실 동일한 코드를 사용했지만, I캐시에는 쓰기 로직이 없었음. 캐시 라인은 2048개고, 4개의 워드가 들어감.

메모리 요청이 들어오면 캐시가 주소에서 워드 오프셋을 제거하고 캐시 인덱스를 계산했음. 캐시 라인은 메모리에서 가져옴. 캐시 라인은 데이터, 태그, 상태로 구성되어 있었음. 태그는 캐시 별칭을 방지했고, 상태 비트는 유효성과 더티 비트를 표시했음. 캐시 라인이 유효하고 태그가 일치하면 캐시에서 직접 읽거나 수정해서 읽었음. 태그가 일치하지 않으면 메모리에서 가져왔음.

캐시 미스일 경우에 대해서 캐시를 초기화하고, 더티 비트가 설정되었을 경우에는 캐시 라인을 먼저 메모리에게 쓰고, 새로운 캐시 라인을 읽어옴.

### 중재자
캐시에서 메모리 요청이 동시에 발생할 경우, 중재자가 이를 해결해 줌. 중재자는 캐시에서 메모리 요청을 받은 후 실제 메모리가 忙하다면 요청을 큐에 넣어두었다가 메모리가闲해지면 요청을 처리해 줌. 이 때, D캐시의 요청을 I캐시의 요청보다 우선순위를 높게 두어서, D캐시의 요청이 먼저 처리되도록 해 줌.

### 마무리
이렇게 해서 우리가 만든 프로세서에서 도둔을 실행하는 것을 성공적으로 완수했음. 하지만 실제 하드웨어에서 돌아가려면 메모리 인터페이스를 더욱 복잡하게 구현해야 할 듯.
