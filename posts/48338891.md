# 해석자들이 biologist 기술 훔치다
GHC에서 -foptimal-applicative-do 플래그가 있는데 이건 기본으로 꺼져 있다가 만약 켜면 ApplicativeDo 최적화를 한다는데 이게 존나 느려서 안쓰는 거임. 하지만 한 해석자가 이게 존나 느리다고? 하고 시작한 이야기.
# ApplicativeDo 는 뭐냐?
do notation에서 sequential.bind를 쓰면 모든 작업이 순차적으로 실행되는데, 만약 작업들 사이에 아무런 의존관계가 없다면 병렬로 처리가 가능하다. 이 병렬성을 표현할 수 있는のが ApplicativeDo 임.
예를들어, 
```haskell
numCommonFriends x y = do
  fx <- friendsOf x
  fy <- friendsOf y
  return (length (intersect fx fy))
```
이 친구들은 그냥 fx, fy를 하나의 네트워크 라운드 트립으로 처리할 수 있음. 근데 이 do notation으로 작성하면 sequential.bind로 바뀌는데, fy가 fx에 의존하지도 않는데 fx 작업이 끝나야 fy 작업을 할 수 있게 되어버림.
## Applicative로 표현

global에 fx, fy가 있는 상황에서 (x fy -> length (intersect fx fy)) <$> friendsOf x <*> friendsOf y 로 작성할 수 있다고 함. 이때 fx, fy는 친구리스트를 가져오는 작업을 의미.
이런식으로 작성하면 fx, fy의 의존관계를 표현할 수 있게됨.
## 컴파일러가 최적화를 하는 방법
컴파일러는 do notation을 Applicative로 최적화 하는데, 이때 작업들 사이의 의존관계를 파악하고, 의존관계가 없으면 <*>을 사용해서 병렬로 처리를 시킴. 만약 의존관계가 있다면 >>=를 사용해서 순차적으로 처리.
이때, 작업들을 그룹화 하는데, 이 그룹화의 비용을 최소화 하는 것이 목표임.
## 동적 프로그래밍
이 그룹화의 비용을 최소화 하는 문제를 동적 프로그래밍을 통해 해결할 수 있다고 함. 이때, RNA折りたたみ를 예로들었는데, 동적 프로그래밍을 사용해서 RNA折りたたみ의 최적의 구조를 찾을 수 있다고 함.
## 최적화 알고리즘
최적화 알고리즘이 O(n³)이라고 하는데, 이 알고리즘이 매우 느리다고 함. 그래서 최적화 플래그를 끄고, 기본 알고리즘으로 사용하고 있다고 함.
## 결론
GHC에서 ApplicativeDo 최적화를 하는데, 동적 프로그래밍의 기술을 사용해서 최적화할 수 있다고 함. 다만, 이 알고리즘이 매우 느리기 때문에, 기본적으로는 끄고 사용한다고 함.
