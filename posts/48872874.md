> **🔗 원문 보러가기:** [We scaled PgBouncer to 4x throughput](https://clickhouse.com/blog/pgbouncer-clickhouse-managed-postgres)


 PgBouncer는 싱글스레드로 돌아감. 그래서 하나의 프로세스가 하나의 CPU 코어만 사용하게 되는데, 16코어 머신이면 하나의 코어가 모든 연결 풀링을 담당하고 나머지 15개 코어가 놀게됨. 결국 연결 풀러가 포스트그레스가 리소스를 다吃기 전에 성능이 제한됨.

클릭하우스 매니지드 포스트그레스에서는 코어 수에 맞춰 PgBouncer 프로세스를 여러개 띄움. 그리고 so_reuseport를 사용해 하나의 포트로 들어오는 요청을 커널이 로드밸런싱하게 함. 그러면 클라이언트는 하나의 PgBouncer만 있는 것처럼 동작하지만, 실제로는 여러개의 프로세스가 함께 작동하고 있음.

 PgBouncer 문서에도 나오지만, so_reuseport로 여러 코어를 사용하는 방법을 설명하고 있음. 하지만 쿼리 취소를 하는데 문제가 있는데, 하나의 프로세스가 쿼리를 처리하는 중에 다른 프로세스가 들어오는 취소 요청을 обработ해야 할 때가 있음. 이 경우 peering을 사용하면 해결됨.

클라이언트 연결은 트랜잭션 모드로 동작해서 서버 연결이 즉시 풀리에 반환이되고, 커넥션 예산은 프로세스마다 분할되어서 포스트그레스가 리소스를 다 먹기 전에 성능이 제한되지 않도록 함.

AWS EC2 인스턴스를 사용해서 성능 테스트를 했음. 하나의 PgBouncer 프로세스를 사용하는 경우와 16개의 프로세스를 사용하는 경우의 성능 차이를 비교했는데, 16개의 프로세스를 사용하는 경우 성능이 4배提高됨. 

클라우드와치의 CPU 사용률 측정 결과도 동일함. 싱글 프로세스의 경우 CPU 사용률이 16%인데, 16개 프로세스를 사용하는 경우 60%임. 결국 하나의 PgBouncer 프로세스는 리소스를 다 못 쓴다는 얘기임.

클라이언트 연결 수도 같은 식임. 싱글 프로세스는 연결 수가 일정 수를 넘으면 새로운 연결을 거부하는데, 16개 프로세스를 사용하는 경우 연결 수가 더 많아도 처리할 수 있음.
   
 결론은 PgBouncer를 여러 프로세스로 사용하고, so_reuseport와 peering을 사용해서 성능을 향상시킬 수 있다는 거임. 클릭하우스 매니지드 포스트그레스에서는 기본으로 이 설정을 사용하고 있으니, 바로 써보기 바람.
