# 포스트GRES 워크플로우는 뭐임?
포스트GRES 워크플로우는 프로그램 상태를 데이터베이스에 저장해서 재실행할 수 있는 기술임. 
이 기술을 사용하면 프로그램이 중간에 터지거나 실패해도 마지막에 저장한 상태에서 다시 시작할 수 있음.

# 일반적인 워크플로우 구현 방법
일반적인 워크플로우 구현 방법은 외부 오케스트레이션 시스템을 사용하는 건데, 
이 방법은 템퍼럴, 에어플로우, AWS 스텝 펑션즈 같은 시스템에서 사용하는 방식임.
이 방식에서는 워크플로우를 여러 단계로 나누고, 외부 오케스트레이터가 각 단계를 조율함.
하지만, 이 방법은 너무 복잡하고, 불필요한 중간자 역할을 수행하는 오케스트레이터가 존재하기 때문에, 
더简单하고 효율적인 방법이 필요함.

# 포스트GRES 워크플로우 시스템
포스트GRES 워크플로우 시스템은 애플리케이션 서버가 직접 포스트GRES와 통신해서 워크플로우를 실행하는 방식임.
클라이언트가 워크플로우를 제출하면, 포스트GRES workflows 테이블에 항목이 생성되고,
애플리케이션 서버는 이 항목을 폴링해서 실행함. 
서버가 워크플로우를 실행할 때, 각 단계의 결과를 포스트GRES에 저장하고,
서버가 다운되거나 실패하면, 다른 서버가 워크플로우를 복구할 수 있음.

이 디자인은 중앙 오케스트레이터가 불필요함을 보여줌.
포스트GRES 테이블의 락킹 클로즈를 사용해서, 
서버가 워크플로우를 dequeuing할 때, 정확히 하나의 워커가 워크플로우를 dequeue할 수 있음.
또한,_worker가 중복 작업을 감지하고, 포스트GRES 데이터베이스完整성 제약조건을 사용해서, 
다중 워커가 동시에 작업을 실행하지 못하게 함.

# 확장성과 가용성
포스트GRES 워크플로우 시스템의 확장성과 가용성은 기본적으로 데이터베이스에 의해 결정됨.
시스템은 작업자 서버를 더 추가해서 수평적으로 확장할 수 있고, 
최대 용량은 데이터베이스가 얼마나 빠르게 워크플로우를 처리하느냐에 달림.
또한, 워커는 взаим대체할 수があり, 서로의 상태를 쉽게 복구할 수 있기 때문에,
시스템은 데이터베이스가 가용한 한 가용하다 할 수 있음.

포스트GRES를 사용하면, 확장성과 가용성에 대해 알려진 솔루션이 이미 존재하기 때문에, 
이점을 활용할 수 있음. 
 확장성을 위해서, 단일 포스트GRES 서버는 수십만 개의 워크플로우를 처리할 수 있고, 
さらに 확장성을 위해서 분산 포스트GRES 또는 샤딩을 사용할 수 있음.
가용성을 위해서, 포스트GRES는 자동 장애조치와 스트리밍 복제를 지원하고, 
관리형 서비스는 높은 가용성 SLA를 제공함.

# 관측 가능성
포스트GRES 워크플로우 시스템에서 워크플로우와 단계를 포스트GRES 테이블에 저장하기 때문에, 
관측 가능성이 내장되어 있음. 
포스트GRES를 사용하면, 거의 모든 워크플로우 관측 쿼리를 SQL로 표현할 수 있고, 
예를 들어, 지난 달에 에러가 발생한 모든 워크플로우를 찾는 쿼리는 다음과 같음:

이 쿼리는 어떤 워크플로우가 에러가 발생했는지 쉽게 찾을 수 있음.
또한, 포스트GRES 테이블에 워크플로우와 단계 데이터를 저장하고, 
보조索引를 추가해서 분석 쿼리를 빠르게 수행할 수 있음.
이렇게 하면, 효율적인 관측 가능성을 얻을 수 있음.

# 신뢰성과 보안
외부 오케스트레이터를 사용할 때, 오케스트레이터와 데이터 저장소가 단일 장애점이 됨.
이 둘 중 하나가 다운되면 전체 애플리케이션이 사용 불가능해짐.
또한, 워크플로우와 단계를 처리하고 저장하기 때문에, 민감한 애플리케이션 데이터에 접근할 수 있기 때문에, 
보안을 강화하고, 접근을 제어하고, 감사를 수행해야 함.

반면에, 포스트GRES 워크플로우 시스템에서는 단일 장애점이 포스트GRES 뿐임.
또한, 모든 워크플로우 데이터는 포스트GRES에 직접 저장되기 때문에, 
또다른 시스템을 거치지 않음.
애플리케이션이 이미 포스트GRES를 사용하고 있다면, 
_durável execution 기능을 사용하면 새로운 단일 장애점이나 보안 취약점을 추가하지 않음.
기존에 중요한 인프라로 사용하는 데이터베이스를 활용하는 것이 더 합리적임.
