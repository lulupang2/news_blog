# C코드 작성 시 컴파일러별로 달라지는 개념들
기초적인 C코드를 작성할 때도 컴파일러에 따라 달라지는 부분이 존재한다는 점은 개발자를 난감하게 만들곤 함. 

## glibc
glibc는 GNU C 라이브러리로, C標準 라이브러리의 구현체 중 하나임. 
이 라이브러리의 헤더 파일들은 컴파일러와 호환성 문제로 인해 개발자를 곤경에 빠지게 함. 
예를 들어, glibc의 `<stdio.h>` 헤더는 컴파일러마다 다음과 같은 문제가 생김:
- `<stdio.h>` 헤더에서 정의된 `struct epoll_event`는 GNU C 컴파일러의 `__attribute__((packed))`를 이용해 정의됨.
- 이는 구조체의 레이아웃을 변경하기 때문에, 컴파일러가 이 확장을 지원하지 않으면 ABI 호환성 문제가 생김.
- 만약 컴파일러가 `__attribute__((packed))`를 지원한다고 해도, glibc의 `<sys/cdefs.h>` 헤더에서 다음과 같이 컴파일러를 체크해 GCC나 Clang이 아니면 이 속성을 무시함.
 ```c
#if !(defined __GNUC__ || defined __clang__ || defined __TINYC__)
#define __attribute__(xyz) /* Ignore */
#endif
```

## SDL
SDL은 Simple DirectMedia Layer의 약자로, 멀티미디어 애플리케이션 개발을 위한 크로스 플랫폼 개발 라이브러리임.
SDL도 컴파일러마다 다른 확장을 지원하거나, 특정 컴파일러에 최적화된 코드가 포함되어 있기 때문에 호환성 문제가 생길 수 있음.
예를 들어, SDL의 `<SDL_endian.h>` 헤더는 바이트 순서 관련 처리를 위한 함수를 포함하고 있는데, 컴파일러 마다 이를 지원하는지에 따라 다르게 처리함.
```c
if (GCC or clang) and __has_builtin(__builtin_bswapX) → use builtins
else if (msvc >= v8.0) -> use msvc intrinsic #pragma
else if defined (ISA-specific macro like __x86_64__) -> use inline assembly
else -> use generic impl with regular bitwise operations
```

## OpenBSD libc
OpenBSD의 libc也是 컴파일러에 따라 다른 확장을 지원하거나, 특정 컴파일러에 최적화된 코드가 포함되어 있기 때문에 호환성 문제가 생길 수 있음.
예를 들어, OpenBSD의 `<signal.h>` 헤더는 다음과 같은 inline 함수를 정의하고 있는데, 이는 C99 표준과 GCC의 동작이 달라 호환성 문제를 일으킴.
```c
__only_inline int sigemptyset(sigset_t *__set) { *__set = 0; return (0); }
```

## 결론
C코드를 작성할 때 컴파일러 마다 다른 확장이나 동작이 있을 수 있기 때문에, 개발을 할 때 컴파일러 호환성을 고려해야 함.
이러한 문제를 해결하기 위해, 컴파일러 개발자는 다음과 같은 방법을 사용할 수 있음:
* 업스트림에서 호환성 문제를 패치함.
* 사용자의 컴파일러에서 특정 확장을 지원하는지 체크함.
* 확장을 지원하지 않는 컴파일러에서 동작하도록 코드를 수정함.
* GCC의 확장을 지원하도록 코드를 수정함.

개인적인 생각으로는, 4번째 방법이 가장 현실적임. 하지만, 이는 컴파일러 개발자들이 자신의 확장을 다른 컴파일러에서 지원하도록 만드는 문제를 해결하지는 못함.
