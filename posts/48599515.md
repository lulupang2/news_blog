   # 인스턴스가 없는 이유 
   이제 알겠지? atPROTO에는 인스턴스가 없음. 인스턴스는 마스토돈식 호스팅+앱이 결합된 것을 말하는 거임:

   atPROTO의 分散化는 '한 앱의 여러 복사본'보다 더 풍부한 구조를 가졌음. 호스팅을 바꾸고 싶으면 바꿀 수 있음. 오늘도自動으로 바꿨음. 이제 유로스카이에 내가 데이터를 올렸기 때문임. 더 대담하면 클라우드플레어에 自由로 데이터를 올릴 수도 있음. 새로운 앱을 만들거나 시도하고 싶으면 만들 수 있음. 탠글드와 심블과 같은 거를 확인해봐. 블루스카이와는 상관없음. 내가 최근에 앱을 만들었고, 오픈소스임. 너도 한 번 만들어봐. 분산화에 대해 관심이 있다면, 여기서 자유롭게 분산화시켜라. 
   #인스턴스 脳에서Escape 
   이제 왜 분산화된 소셜 미디어 토론이 항상 이렇게 되는지 알겠지? 마스토돈 유저들은 인스턴스의 수로 분산화를 측정하지만, 마스토돈에서는 이것밖에 못 함. '박스'는 한 종류뿐이고, 각 박스는 '앱+호스팅'임. 그러면 박스를 여러 개 복사해서 서로 통신하게 하는 것밖에 못 함. 
   atPROTO에서 앱은 모두 블루스카이의 투영된 거임. rss와 구글 리더가 블로그스피어의 투영된 거랑 비슷한 거야. 분산화를 하기 위해서는 호스팅을 바꾸거나, 새로운 앱을 만들거나 하는 거임. 블루스카이 데이터베이스 서버를 여러 개 복사하는 건 가능지만, 구글 리더를 여러 개 복사하는 것만큼 쓸모는 없음. 
   사람들이 필요에 따라서 셋팅하긴 함. 블랙스카이 같은 걸 보면 답이 나옴. 이런 접근법은 여러 가지인데, 이 블루스카이 클라이언트는 전용 데이터베이스가 없이, 커뮤니티에서 제공하는 호스팅 캐시를 사용하기도 함. 
   릴레이 같은 공유 네트워크 인프라는 1년 전부터 운영하기廝易해졌음. 
   이래서 블루스카이 인스턴스를 셀 수 있는 건誤導性 있음. 
   중요한 건, 사람들이 대체 호스팅으로 이동하고 있는가? 새로운 앱을 만들고 시도하는가? 
   호스팅과 앱을 분리하면, 폐쇄적이고 연합된 소셜 미디어의 분산화된 장단점을 고칠 수 있음.
