### 포스트그레스 19에期待되는 쿼리 힌트 기능
포스트그레스 19에 새로운 기능이 추가되었는데, 그것은 바로 쿼리 힌트 기능임. 근데 이름이 hints가 아니고 advice로 바뀌었다는 점이有点 감동임. 
포스트그레스 커뮤니티에서는 오랜 시간 동안 쿼리 힌트에 대해 매우 부정적인 견해를 가지고 있었다고 함. 공식 위키 페이지에는 쿼리 힌트가 문제가 되는 여섯 가지 이유가 제시되어 있음. 
그리고 이에 대해 로버트 하스라는 사람이 반론을 제기했는데, 그는 쿼리 힌트가 필요하다고 주장하며, 다음과 같이 말했음.
> 우리는 쿼리 힌트를 원하고, 그것이 실제로 작동하는 것을 원한다. 그리고 그것이 쓰레기 되지 않기를 원한다.
 том 레인은 다음과 같이 말했음.
> 나는 힌트가 문제가 되지 않는 시스템을 본 적이 없다... 하지만 그것이 불가능하다는 것을 의미하지는 않는다.
 쿼리 힌트에 대한 논의는 2010년부터 시작되어 여전히 진행 중임. 그리고 그 결과로 pg_plan_advice와 pg_stash_advice라는 두 가지 새로운 contrib 모듈이 추가되었음.
 pg_plan_advice는 쿼리 힌트를 제공하여 쿼리 최적화를 위한 조언을 주는 모듈임. 그리고 pg_stash_advice는 이 모듈을 사용하여 쿼리 조언을 저장하고 관리하는 모듈임.
 이 기능을 사용하면 쿼리를 최적화하는 데 도움이 되는데, 특히 복잡한 쿼리나 성능이 좋지 않은 쿼리에 유용함. 
 예를 들어, 다음과 같은 쿼리가 있을 때
 ```sql
SELECT * FROM my_fact f JOIN my_dim d ON f.dim_id = d.id;
```
 쿼리 힌트를 사용하여 쿼리 계획을 조정할 수 있음. 그리고 이 힌트는 쿼리 자체에 포함되지 않고, 별도의 설정으로 관리됨.
 ###(pg_stash_advice 사용법)
 pg_stash_advice를 사용하려면 다음과 같이 설정해야 함.
 ```sql
CREATE EXTENSION pg_stash_advice;
```
 그리고 다음과 같이 사용하면됨.
 ```sql
SET pg_plan_advice.advice = 'INDEX_SCAN(f idx_fact_dim_id)';
```
 그리고 다음과 같이 사용할 수도 있음.
 ```sql
CREATE TABLE my_fact ( id BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY, dim_id BIGINT NOT NULL );
CREATE TABLE my_dim ( id BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY, stuff TEXT );
CREATE INDEX idx_fact_dim_id ON my_fact (dim_id);

EXPLAIN (COSTS OFF, PLAN_ADVICE) SELECT * FROM my_fact f JOIN my_dim d ON f.dim_id = d.id;
```
 이러면 다음과 같은 결과가 나옴.
 ```
QUERY PLAN 
----------------------------------
 Hash Join  (cost=10.70..20.70 rows=10 width=24)
   Hash Cond: (f.dim_id = d.id)
   ->  Seq Scan on my_fact f  (cost=0.00..10.70 rows=10 width=12)
   ->  Hash  (cost=10.70..10.70 rows=10 width=12)
         ->  Seq Scan on my_dim d  (cost=0.00..10.70 rows=10 width=12)
 Generated Plan Advice: 
 JOIN_ORDER(f d) 
 HASH_JOIN(d) 
 SEQ_SCAN(f d) 
 NO_GATHER(f d)
```
 이러면 이 결과를 사용하여 쿼리 힌트를 설정할 수 있음.
 이 기능을 사용하면 쿼리를 최적화하여 성능을 개선할 수 있고, 특히 복잡한 쿼리나 성능이 좋지 않은 쿼리에 유용함.
