### HTTP 쿼리 메소드登場!
이새끼가 뭔데?  HTTP 쿼리 메소드(RFC 10008)가 공식적으로 등장했단다. 이 메소드는 **GET**과 **POST**의 단점을 보완해서 나왔음.

**GET**은 URI를 이용해 쿼리를 요청하는 방식인데, URI의 길이 제한에 걸리기 쉽고, URI를 로깅하기 쉬워 보안에 취약한 점이 있음. 
**POST**는 본문에 쿼리를 담아서 요청하는 방식인데, 본문을 처리하는 서버가 안전하고 멱등해야 한다는 보장이 없다는 점이 있음.

그렇다고 **GET**이나 **POST**를 쓰지 말라는 건 아님. 단지, 쿼리를 요청할 때 **GET**의 단점을 보완하면서 **POST**의 본문 처리 특성을 가지는 ** 쿼리 메소드**를 사용할 수 있을 뿐임.

### HTTP 쿼리 메소드란?
쿼리 메소드는 서버에게 본문에 담긴 쿼리를 처리하라고 요청하는 메소드임. 이 때, 쿼리는 본문에 담기 때문에 URI의 길이 제한에 걱정할 필요가 없음. 또한, 쿼리 메소드는 **안전**하고 **멱등**하기 때문에 캐싱이나 자동 재시도가 가능함.

쿼리 메소드의 본문은 **쿼리 내용**을 담고, 쿼리 내용의 형식은 **미디어 타입**에 의해 결정됨. 쿼리 메소드는 **안전**하고 **멱등**하기 때문에, 캐싱이나 자동 재시도가 가능함.

### HTTP 쿼리 메소드의 장점
HTTP 쿼리 메소드는 **GET**과 **POST**의 단점을 보완한 메소드임. 쿼리를 요청할 때, URI의 길이 제한에 걱정할 필요가 없고, 쿼리 내용의 형식은 미디어 타입에 의해 결정됨. 또한, 쿼리 메소드는 **안전**하고 **멱등**하기 때문에 캐싱이나 자동 재시도가 가능함.
