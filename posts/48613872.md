### 리눅스 epoll vs io_uring 대첩

저는 작년에 학생들과 함께 리버스 프록시 서버를 만들었음. TinyGate라고 이름 붙였는데, 당시에는 그냥 간단하게 만들었고,.worker 기반으로 작동해서 괜찮았음. 하지만 성능은 기대할 수 없는 niveau였다는 거임. 그 때_students들은 실망했었는데, 그들이랑 같이 epoll랑 io_uring에 대해研究를 하게 됨.

결론부터 말하면, 우리는 tinyGate를 epoll 기반으로 처음 만들었지만, 나중에 io_uring으로 chuyển換했음. 그리고 그 결과, 성능이 ㄹㅇ 개선됨.

### epoll의 역사

epoll은 리눅스에서 async I/O를 처리하기 위해 만들어졌음. 2002년도에 리눅스 커널에 추가되었는데, 당시에는 유일한 선택지였음. 하지만, epoll은 syscalls를 많이 사용함. 예를 들어, I/O가 가능할 때通知를 받은 후에, 실제 I/O 작업을 해야 하는데, 이때마다 syscalls를 호출해야 함. 그리고, 각 syscalls마다 context switch가 일어나는데, 이게 바로 오버헤드의 원인이 됨.

### io_uring의 등장

그렇다 보니, 2019년에 io_uring이 등장했음. io_uring은 epoll보다 더 좋은 async I/O 처리 방법임. io_uring은 I/O 작업이 완료될 때通知를 받는 대신에, I/O 작업을 할 때 바로通知를 받을 수 있음. 그리고, io_uring은 kernel과 user space 사이에 shared memory를 사용해서 syscalls를 줄일 수 있음. 예를 들어, I/O 작업을 submission queue에 넣으면, kernel이 자동으로 처리를 함. 그리고, completion queue에서 결과를 받을 수 있음.

### epoll vs io_uring

기본적으로, epoll은 I/O 작업이 가능할 때通知를 받고, io_uring은 I/O 작업이 완료될 때通知를 받음. 그리고, epoll은 syscalls를 많이 사용하지만, io_uring은 shared memory를 사용해서 syscalls를 줄일 수 있음.

### 코드 비교

```c
// epoll 예시
#include <stdio.h>
#include <unistd.h>
#include <sys/epoll.h>
#include <stdlib.h>

int main() {
    // ...
    int epoll_fd = epoll_create1(0);
    // ...
    struct epoll_event ev, events[MAX_EVENTS];
    ev.events = EPOLLIN;
    ev.data.fd = STDIN_FILENO;
    // ...
    int n = epoll_wait(epoll_fd, events, MAX_EVENTS, -1);
    // ...
    for (int i = 0; i < n; i++) {
        if (events[i].data.fd == STDIN_FILENO) {
            char buf[256];
            ssize_t count = read(STDIN_FILENO, buf, sizeof(buf));
            printf("read %zd bytes\n", count);
        }
    }
    // ...
}

// io_uring 예시
#include <stdio.h>
#include <unistd.h>
#include <liburing.h>
#include <stdlib.h>

int main() {
    // ...
    struct io_uring ring;
    char buf[256];
    // ...
    if (io_uring_queue_init(8, &ring, 0) < 0) {
        // ...
    }
    // ...
    struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
    io_uring_prep_read(sqe, STDIN_FILENO, buf, sizeof(buf), 0);
    // ...
    io_uring_submit(&ring);
    // ...
    struct io_uring_cqe *cqe;
    if (io_uring_wait_cqe(&ring, &cqe) < 0) {
        // ...
    }
    if (cqe->res < 0) {
        // ...
    } else {
        printf("read %d bytes\n", cqe->res);
    }
    // ...
}
```

### 정리

요약하자면, io_uring은 리눅스에서 async I/O를 처리하는 새로운 방법임. epoll보다 더 좋은 성능을持ち고, shared memory를 사용해서 syscalls를 줄일 수 있음. 그리고, io_uring은 I/O 작업이 완료될 때通知를 받을 수 있어서, 더 효율적인 async I/O 처리를 할 수 있음.

결론은, 리눅스에서 async I/O를 처리할 때, io_uring을 사용하는게 더 좋음. 하지만, epoll도 여전히 쓰이고 있고, io_uring을 사용하는 것에 익숙해지는데 시간이 걸릴 수 있음. 그러니까, 그냥 둘 다 잘 알아둬서 상황에 맞게 쓰면 됨.
