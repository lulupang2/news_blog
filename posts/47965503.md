### F#로 게임보이 에뮬레이터를 만들었음 ㅋㅋㅋㅋ 
   요즘 소프트웨어 엔지니어로 일하는지는 8년이 넘었지만 컴퓨터가 실제로 어떻게 동작하는지는 잘 몰랐음. 
   그래서 컴퓨터가 어떻게 동작하는지 배우기 위해 에뮬레이터를 만들기로 했음. 
   벤 이터, 미안하지만 일단 하드웨어를 직접 만들지는 않겠음 ㅋㅋㅋㅋ. 
  子供 때 포켓몬을 잡느라 수 trăm 시간을 GAME BOY를 잡고 있었음. 
   게임보이는 실제 하드웨어이고, 상대적으로 단순한 스코프고, 개인적으로 감정적인 연결이 있는 대상이였음. 
   일단 에뮬레이터를 만들기 전에 From NAND to Tetris라는 강의를 들었음. 
   그 강의는 컴퓨터의 기본적인 원리를 정말 잘 설명해줬고, 레지스터, 메모리, ALU에 대한 이해를 sâu고깊은 것이라고 생각함.

   어쨌든 게임보이 에뮬레이터가 동작하는 단계까지 만들어 낸 것 같음.

   ### 내부 동작 원리
   이 에뮬레이터는 데스크탑과 웹에서 동작해야 했기 때문에, 에뮬레이터 코어와 프론트엔드 사이의 간단한 인터페이스를 만들음. 
   프론트엔드와 코어 사이의 인터페이스는 두 개의 배열과 두 개의 함수로 이루어져 있음. 
   - `framebuffer`: 160x144 픽셀의 배열로, 각 픽셀은 흰색, 밝은 회색, 어두운 회색, 검은 색 중 하나임. 
   - `audiobuffer`: 32768 Hz 샘플 레이트의 링 버퍼로, 읽기와 쓰기 포인터가 있음. 
   - `stepEmulator()`: CPU 명령어 하나를 실행하고, 실행에 걸린 사이클 수를 반환함. 
   - `getJoypadState(state)`: 프론트엔드가 에뮬레이터에 조이패드 상태를 알려주는 콜백 함수로, 보통 프레임당 한번 호출됨. 
   게임보이의 실제 하드웨어와 비슷하게 Fame Boy를 모델링했음. 
   CPU는 실제 샵 LR35902와 마찬가지로, 메모리 맵과 인터럽트 신호만을 알게 되어 있음. 
   이 부분은 내가 코드베이스에서 가장 F#다운 부분이라고 생각함. 
   `Memory.fs`에는 게임보이에서 사용되는 대부분의 RAM이 담겨 있고, 메모리 맵과 버스를 CPU, IO 컨트롤러, 카트리τζ 사이에서 제공해줌. 
   또한 PPU와 같은 VRAM과 OAM RAM 배열을 공유해줌. 
   `IoController.fs`는 메모리 파일에 로직을 너무 많이 추가해서 만들어짐 ㅋㅋㅋㅋ. 
   게임보이 하드웨어에는 IO 컨트롤러가 따로 없지만, IO 컨트롤러를 통해 하드웨어 레지스터를 다루면 코드가 간결해지고 안전해짐. 
   `Emulator.fs`의 `stepper` 함수는 에뮬레이터의 모든 구성 요소를 하나로 묶어줌. 
   ```fsharp
let stepper () = 
    // CPU 명령어 하나를 실행하고, 실행에 걸린 사이클 수를 반환함. 
    let mCycles = stepCpu cpu io 
    for _ in 1..mCycles do 
        stepTimers timer io 
        stepSerial serial io 
    // APU는 CPU보다 4배 빠르지만, 배치 처리할 수 있음. 
    let tCycles = mCycles * 4 
    // PPU는 CPU보다 4배 빠르고, APU도 여기 있으면 좋겠음. 
    for _ in 1..tCycles do 
        stepPpu ppu 
    // 사이클 수를 반환해서 프론트엔드가 에뮬레이터를 올바른 속도로 실행하도록 함. 
    mCycles
   ```
   ### CPU와 F# 
   일단 F#에 대한 순수한 프로그래머들에게 미안하다고 말하고 싶음. 
   내 CHIP-8 에뮬레이터는 완전히 순수한代码지만, Fame Boy는 변이성(mutability)을 많이 사용해버림 ㅋㅋㅋㅋ. 
   게임보이는 CHIP-8보다 훨씬 빠르고, 16KB가 넘는 메모리를 сек당 수百만 번 복사하는 것은 좋은 fikir이 아니라는 생각에 이런 선택을 했음. 
   그럼에도 불구하고 F#를 사용한 이유는 두 가지임. 
   첫째, TYPE 시스템이 CPU 명령어를 모델링하는 데 정말 잘 맞음. 
   둘째, F#를 정말 좋아함. 이전 회사에서 F#로主要로 일했어서, 언제나 F#를 계속 사용할 방법을 찾고 있음 ㅋㅋㅋㅋ. 
   ### 도메인 모델링 
   CPU 명령어를 모델링하는 데 F#가 잘 맞는 이유를 설명하기 위해, Gekkio의 Complete Technical Reference를 보면서 CPU를 구현했음. 
   명령어를 분류하고, 다음과 같은 코드를 작성했음. 
   ```fsharp
type LoadInstr = 
    | Load8Immediate of uint8 
    | Load8Direct of Register 
    | Load8Indirect 
    // ... 
type ArithmeticInstr = 
    | IncrementDirect of uint8 
    | IncrementIndirect of Register 
    // ... 
   ```
   이렇게 하면 CPU 명령어가 512개가 아닌 58개로 줄어듬. 
   ### 테스팅 
   일단 CPU를 테스트하기 위해, 단순히 다음과 같은 코드를 작성했음. 
   ```fsharp
match opcode with 
| 0x00 -> Nop 
| _ -> failwith ""Unimplemented opcode" 
   ```
   그리고 테트리스 ROM을 실행해서 테스트했음.
