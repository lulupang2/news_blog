### ARM 어셈블리로 웹서버 짜보기
 ARM 어셈블리로 웹서버를 만들면 뭘 하나요? 그냥 멋있어 보이니까요 ㅋ. 
 어쨌든 macOS용的小小한 정적 HTTP 웹서버인 ymawky를 ARM 어셈블리 언어로 만들어서 공유하고 싶었음. 
 raw Darwin 시스템콜을 사용했고, 정적 파일들은 지원해 주고, GET, HEAD, PUT, OPTIONS, DELETE 요청도 다 지원해 줌. 
 또한 바이트 범위, 디렉토리 목록, 커스텀 에러 페이지도 모두 지원해 줌. 
 왜 이런 짓을 했냐고요? 그냥 하는 거임. 
 저는 시스템/_low-level_ 배경을 가지고 있어서, 웹서버가 실제로 어떻게 동작하는지에 대해서는 잘 모르겠다는 생각이 들었음. 
 그래서 Assembly 언어로 웹서버를 만들면서, 관련된 문제들에 대해서 학습해 보았음. 
 물론 이게 nginx를 대체하지는 못할 뿐더러, 가장 어려운 방법으로도 작동하긴 하지만... 
 ymawky는 fork-on-request 서버로, 새로 들어오는 연결마다 fork() 시스템콜을 호출하도록 설계하였음. 
 이러면 메모리가 공유되지 않아서, 이해하기도 쉽고, 작성하기도 쉽，不过也有很多 단점이 있다고 생각함... 
 프로세스당 메모리 공간이 따로 존재하고, 근본적으로는 nginx처럼 event-driven async non-blocking 모델보다 동시적인 연결 처리가 부족하다는 것. 
 그리고 메모리 사용도 매우 많음. 
 소켓에 바인딩하고 리스닝하는 건 쉽，不过 요청을 처리하는 건 정말 힘든 일임. 
 요청 유형을 결정하고, 요청한 경로를 추출하고, 경로를 정규화하고, 안전성 검사를 수행하고, 헤더 필드를 파싱해야 함. 
 또한 요청한 파일에 대한 정보를 얻고, 파일이 디렉토리인지 일반 파일인지 결정해야 하고, 업로드된 본문을 임시 파일에 기록해야 함. 
 응답 헤더를 작성하고, 응답 본문을 작성하고, 열린 파일을 닫고, 오류를 처리해야 하는데... 
Assembly 언어로 웹서버를 작성하면, HTTP 라이브러리가 없다는 단점이 있음. 
 자동으로 리소스를 정리해 주는 것도 없고, 문자열 타입도 없는데, 문자열은 그냥 메모리 단위의 바이트 시퀀스일 뿐임. 
C 에서 struct 는 언어의 특징이지만, Assembly 언어에서는 메모리 오프셋을 정확하게 계산해야 함. 
yamwky는 raw 시스템콜을 사용하기 때문에, 커널에 직접 호출을 하게 되고, 이건 매우 어려운 일임... 
 요청을 받으면, 요청 유형을 먼저 결정하고, 요청한 경로를 추출해야 함. 
 헤더를 처리하는 건 정말 싫은데, 특히 문자열 파싱은 너무 싫다고 생각함... 
 요청에 대한 처리는 정말 힘든 일이지만, 그래도 Assembly 언어로 웹서버를 만들었다는 자신감은 있음. 
 그리고 PUT 요청은 idempotent 하다는 특징이 있는데, 요청의 결과가 동일하다는 뜻임. 
 따라서 PUT 요청에 대해서도 처리를 해야 함. 
요약하자면 ARM 어셈블리 언어로 웹서버를 만들었다는 거임. 그리고 이건 정말 어려운 일이었음. 하지만 그래도 자신감을 가지고, 더 많은挑戰을 해 보고 싶다고 생각함.
