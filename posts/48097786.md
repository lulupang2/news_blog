### 마이크로소프트 3D 무비 메이커, 리눅스에 떴다!!
   요즘은 리눅스도 게임할 수 있지 하지만, 리눅스에서 돌아가는 구닥다리 멀티미디어 소프트웨어는 찾아보기 힘들지 않음? 
   그래서 오늘은 마이크로소프트가 깜짝 공개한 소식인데, 리눅스용 3D 무비 메이커 소스 코드가公開됐어! 이제 리눅스에서도 3D 애니메이션 영화를 만들 수 있당!
   ### 배경
   2020년에 저는 마이크로소프트 3D 무비 메이커의 역공학에 대한 블로그 시리즈를 썼었지. 그때는 아직 3D 무비 메이커의 원본 소스 코드가 어떻게 생겼을까 하는 생각밖에 못 했었는데, 2022년 5월에 마이크로소프트가 이 소스 코드를 공개하는 큰 일이 발생했어! 
   그리고 Alice Averlong(전 Foone Turing)씨의 끈기 있는 노력과 Scott Hanselman, Jeff Wilcox, 그리고 마이크로소프트 팀의 도움으로 이 소스 코드가 공개될 수 있었어. 소스 코드 공개의 과정에 관심이 있다면 Hanselminutes 팟캐스트를 들어봐. 
   이 깜짝 공개에는 3DMM 애플리케이션(코드명 Socrates), 3DMM과 크리에이티브 라이터 2에서 사용하는 Kauai 애플리케이션 프레임워크, 개발 및 저작工具(예를 들어 이전에 내가 역공학한 스크립팅 언어용 컴파일러), 일부 문서 및 사전 렌더링된 에셋이 포함되어 있어. 이 공개는 충분히 3DMOVIE.EXE를 自分で 컴파일해서 돌릴 수 있을 정도로 완전해! 
   공개되자마자, 저는 소스 코드를 뒤져보았고, 원본 개발 도구로 빌드하는 방법에 대한 노트를 쓰기도 했어. 코드베이스를보니-engineering이 정말 잘되어 있었어. 코드 스타일이 일관적이고, 코드는 잘 주석처리되어 있었고, assert 체크도 곳곳에 있었어. 1995년에 개발한 코드치고는 정말 깨끗했어. 
   ### 3D 무비 메이커 포팅하기
   첫 생각은 새로운 플랫폼으로 포팅해 보는 것이었어. 하지만 포팅을 하려면 너무 많은 작업이 필요하다고 생각해서, 3DMMForever 프로젝트에 加入했어. 이 프로젝트는 3DMM를 現代化하는 것을 목표로 하고 있었기 때문이야. 
   우리가replace한 오래된 makefile을 CMake로 교체했고, Visual Studio 2022와 함께 컴파일하는 문제를 해결했어. 이것은 3DMM를 다른 플랫폼으로 포팅하는 첫 걸음이었지. 
   하지만 프로젝트가 잠시 중단되더니, 2024년 말에 3DMMForever를 포크해서 3DMMEx를 만들었어. 원래 목표는 3DMMForever의 작은 문제를 고치고, 다른 플랫폼으로 포팅하기 위해 필요한 것을 탐색하는 것이었지. 하지만 포팅을 하는데 어려움이 많았어. 
   코드가 pre-표준화된 C++방언을 사용해서, 오직 마이크로소프트 비주얼 C++에서만 컴파일되더라고. 
   Kauai 프레임워크는 크로스 플랫폼을 지원하는데, 정확하게는 두 개의 플랫폼을 지원해. 윈도우와 맥ิน토시 68K야. 하지만 맥인토시 지원은 아주 불완전해. 
   3DMM 애플리케이션은 Kauai의 크로스 플랫폼 추상을 기반으로 빌드되었지만, 때때로 윈32 API를 직접 호출하기도 해. 
   프레임워크에는 인라인 x86 어셈블리 언어가 들어가 있었어. 퍼포먼스에 민감한 함수에서 사용되더라고. 
   프로젝트에는 포인터 크기에 대한 가정들이 있었어. 64비트 시스템에서 컴파일 에러를 발생시키는 거야. 
   프레임워크에는 3DMM에서 사용되지 않지만, 저작도구에 필요한 많은 기능들이있었어. 
   외부 종속성은 pre-컴파일된 정적 라이브러리로 빌드에 연결되어있었어. 
   대부분의 코드에는 테스트가 없었어. 3DMM 개발자들은 자동化된 테스트를 사용했지만, 이 테스트는公開되지 않았어. 
   3DMM의 소스 코드를 그대로 받아서 다른 플랫폼에서 컴파일을 시도하면, 동시에 모든 문제를 직면해야 했을 거야. 그래서 나는 문제를 분리하여 해결하는 방식을 선택했어. 
   그리고 Mark Cave-Ayland라는 개발자와 협력해서 3DMM를 리눅스로 포팅하기 시작했어. Mark은 3DMM를 리눅스로 포팅하여라즈베리 파이에서 실행하고 싶었던 거지. 
   ### 정적 라이브러리
   3DMM 프로젝트에는 두개의 외부 종속성이 정적 라이브러리形式으로 패키징되어있었어. BRender와 AudioMan이それ야. 
   BRender는 Argonaut Technologies에서 개발한 3D 렌더링 엔진이야. 3DMM 소스 코드 공개와同時に BRender의 소스 코드도 공개되었어.하지만 BRender에는 많은 수의 x86 어셈블리 언어 코드가 들어가 있었어. 
   AudioMan은 1990년대 중盤에 마이크로소프트 멀티미디어 제품에서 사용된 사운드 믹서 라이브러리야. 3DMM에서는 사운드 재생뿐만 아니라 가져온 녹음된 사운드를변환하는 데 사용되어. 
   정적 라이브러리는 윈도우 x86과 비주얼 스튜디오에서 컴파일할 때만 동작했어. 그래서 저는 AudioMan을 역공학해서 C++로 디컴파일하기 시작했어. 
   디컴파일은 обычно bastante 어렵지만, 이 경우에는 디버그 빌드에서 전체 기호 정보를가져올 수 있었어. 3DMM에서 동작하기 위해 필요한 코드만 디컴파일하면 됐기 때문에, 완전한 디컴파일은필요 없었어. 
   AudioMan을 디컴파일하는 것은 윈도우 x64와 ARM64 버전의 3DMMEx를 빌드하는데 중요한 부분이었어.하지만 AudioMan은 윈도우 사운드 API와 COM 인터페이스에 꽉 잡혀 있었기 때문에, 비윈도우 플랫폼에서작동이 쉽지 않았어. 그래서 비윈도우 플랫폼에서는 miniaudio를 이용한 새로운 波音 재생 모듈을 추가했어. 
   miniaudio를통해 비윈도우 플랫폼에서 사운드 INPUT도 구현했어. 
   ### 어셈블리 언어除去
   Kauai의 일부 함수에는 핸드최적화된 어셈블리 언어 코드가 들어가 있었어. 퍼포먼스에 민감한 함수에서 사용되었어. 
   컴파일타임 #define을 사용해서 C++와 어셈블리 언어 구현을 переключ할 수 있었어. 일부 함수에는 인텔 x86과 모토로라 68020 용으로 최적화된 버전이있었어. 
   어셈블리 언어 버전을 제거하는 것은 간단했어. #define을지우기만 하면돼. 하지만 C++ 구현에 버그가 몇 개 있음을 발견했어. 어셈블리 언어 버전을 제거하는 것은 퍼포먼스를 조금 cải善했어. 
   이유는 앱이 现代 프로세서에 최적화된 C 런타임의 memcpy/memmove実装을 사용할 수 있었기 때문이야. 
   ### 기타 工程作業
   3DMM을 새로운 플랫폼으로 포팅하기 위해서는, 기존 코드를 많이 수정해야 해. 그래서 코드가 깨지지 않도록, 디버깅과 테스트의 환경을 cải善하기로 했어. 
   Kauai에는 작은 단위 테스트가 몇 개 있었어. 저는 이 테스트를 Google Test로이전해서 빌드에 통합했어. 
   Kauai는 디버거에서 읽기 어려운 커스텀 데이터 타입을 사용했어.，所以 저는 Visual Studio에서 커스텀 데이터 타입을 렌더링하기 위해 NatVis 시각화를 추가했어. 예를 들어, 문자열 클래스(STN)는 이제 문자열 값을表示하기 시작했어. 
   파일 형식 호환성을 깨뜨리지 않도록 주의했어. 그래서 파일 형식 구조체가 다른 플랫폼에서 크기가 변경되지 않도록 정적 assert를 추가했어. 64비트 시스템에서 3DMM을 포팅할 때, 이것은 많은 문제를 발견하는 데 도움이 됐어. 
   UI의 논리를 디버깅하는 것이 어려웠어. 왜냐하면 대부분의 논리가 스크립트안에 들어가 있었기 때문이야.所以 스크립트의 실행을 더 쉽게 추적할 수 있도록 추가 로깅을 더했어. 
   저는 여전히 원래의 Win32 구현을 제거하지 않은 채로, 새로운 구현을 추가할 수 있는 세움을 추가하기 위해 여러 작은 리팩토링을했어. 
   ### 결론
   요약하자면, 이제 3D 무비 메이커를 리눅스에서 직접 돌릴 수 있게 됐어! 마이크로소프트가 깜짝 공개한 소스 코드 덕분에 가능成了어.
