# ORM은 그냥 SQL 배우는 거랑 똑같음
지난 30개월 동안 나는 PostgreSQL과 SQLite를 사용해야 했음. 대부분의 작업은 SQLAlchemy와 Hibernate를 사용해서 했는데, SQLAlchemy는 나쁘지 않았지만 Hibernate는 정말 싫어함. 기존 코드와 데이터 모델을 사용하기도 했고,自ら 모델을 설계하기도 했음. 대부분의 데이터는 이벤트 기반 저장소로, 보고서를 생성하는 데 중점을 두고 있었음.

많은 사람들이 Object/Relational Impedance Mismatch에 대해 이야기했음. 이 문제를 겪어보지 않으면 이해하기 어려운 문제임. Neward의 유명한 논문에서 ORM의 한계를 잘 설명했는데, 나는 직접 그런 문제를 겪어봤음. 엔티티 아이덴티티 문제, 쿼리 최적화 문제, 부분 객체 문제 등이 있었음.

## ORM의 문제점
Partial objects, attribute creep, and foreign keys
ORM의 가장 문제가 되는 점은 'attribute creep'이라고 부르는 문제임.즉, 테이블이 점점 더 많은 속성을 추가하게 되는 문제임. 예를 들어, 클라이언트가 보고서에 다양한 비즈니스 논리를 적용하는 데 필요한 데이터를 제공할 수 있음. 그런데, 그 데이터에 대한 통찰력이 부족해서, 그냥 데이터를 옮겨 담기만 하는 경우가 있음. 이렇게 테이블이 커지면, ORM으로 쿼리를 작성할 때 문제가 생김. ORM은 SELECT *를 사용하게 유도하는데, 이는 실제로 필요한 데이터보다 더 많은 데이터를 가져오게 됨.

## 데이터 가져오기
쿼리 최적화는 ORM에서 정말 힘든 문제임. ORM이 실제로 어떤 SQL을 생성하는지 모르는 경우가 많음. 그래서, 실제로 실행되는 SQL이 효율적인지 확인하기 어려움. 효율적인 SQL을 작성하기 위해서는, 직접 SQL을 작성해야 하는 경우가 많음. 하지만, ORM을 사용하면, SQL을 작성하는 대신에, ORM의 쿼리 언어를 사용해야 함.这样하면, 개발자는 SQL의 세부 구현을 신경 쓸 필요가 없지만, 효율적인 쿼리를 작성하기가 더 어려워지게 됨.

## Duel Schema 문제
Duel Schema 문제는 ORM에서 흔히 나타나는 문제임.즉, 데이터베이스와 애플리케이션에 두 가지 서로 다른 스키마가 존재하는 문제임. 만약, 데이터베이스의 스키마를 애플리케이션에 직접 반영한다면, 데이터베이스의 스키마가 변경될 때마다 애플리케이션도 변경되어야 함. 하지만, 애플리케이션의 스키마를 데이터베이스에 반영한다면, 데이터베이스에 쿼리를 날릴 때마다 스키마르 확인해야 함.

# ORM의 한계
ORM은 개발자가 SQL을 잘 모를 때, 편리하게 데이터베이스를 사용할 수 있는 방법임. 하지만, 실제로 데이터베이스를 잘 사용하기 위해서는, SQL을 잘 알아야 함. ORM은 개발자가 데이터베이스를 사용할 때, 편리함을 제공하지만, 효율성을 제공하지는 못함. 그래서, ORM을 사용할 때는, 개발자가 데이터베이스의 구조와 SQL을 잘 알아야 함.

결론: 그냥 SQL을 배우세요. ORM은 SQL을 대신하지 못함. ORM은 개발자가 데이터베이스를 사용하기 쉽게 해주지만, 효율성을 제공하지는 못함. 그래서, 데이터베이스를 잘 사용하기 위해서는, 개발자가 SQL을 잘 알아야 함.
