# Foldkit: 정확한 프론트앤드 프레임워크 

     ## 소개 

     Foldkit은 Effect와 Elm 아키텍처를 기반으로 하는 프론트엔드 프레임워크야. 이 프레임워크는 타입스크립트로 작성되어 있고, npx create-foldkit-app@latest로 쉽게 설치할 수 있대. 

     ## 핵심 기능 

     - 선언적 상태 관리: 전체 애플리케이션 상태를 하나의 불변 모델로 관리할 수 있어. ogni 변경이 단일 업데이트 함수를 통해 흐르고, 숨겨진 변이或者 staleness가 없어. 
     - 명시적 효과:副 효과는 핸들러에 묻혀있는 임시적인 호출이 아니야. 명령어가 효과를 반환하고, 런타임이 언제, 어떻게 처리할지 결정해. 

     - 규모와 그레이스: 복잡성이 선형적으로 증가해. 50개의 파일이 있는 앱은 5개의 파일이 있는 앱과 같은 패턴을 따르니까, 새로운 팀원들이 코드를 쉽게 이해할 수 있대. 

     ## 아키텍처 

     Foldkit은 Effect를 기반으로 하고, 모든 상태는 단일 스키마로 모델링되어 있고, 부작용은 실패하지 않는 효과로 모델링되어 있어. 

     ## 테스팅 

     Foldkit은 두 가지 테스트 원시값을 제공해. Story는 메시지를 업데이트 함수로 보냈을 때 결과 모델과 명령을断言할 수 있고, Scene은 렌더링된 뷰를 통해 기능을 테스트할 수 있대. 

     ## 개발자 도구 

     Foldkit은 개발자 도구를 제공해. 메시지, 모델 상태, 명령을inspect할 수 있고, 시간을 거슬러가면서 UI를 과거의 모델로 되돌릴 수 있어. AI 에이전트도 MCP를 통해 연결할 수 있대. 

     ## 비교 

     Foldkit은 React, Vue, Svelte, Solid 등과 비교했을 때, 프레임워크가 아키텍처를 규정한다는 점이 다르대. 

     ## 대상 

     - Effect开发자: 同じ 에코시스템, 同じ 패턴, コンテキスト 스위치가 없어. 
     - 정확성을 重視하는 개발자: 아키텍처가 버그를 방지해주니까. 
     - 팀원들이 일관된 패턴을 유지해야 하는 경우: 상태, 효과, 뷰에 대한 일관된 패턴이 있어서, 설움이 줄어드는데. 
     - 복잡한 상태를 가진 프로젝트: 인증 흐름, 실시간 데이터, 멀티 스텝 폼 등이 명시적인 상태와 전환으로 관리되니까. 
     ## 비대상 

     - 대규모 기존 React 코드베이스: Foldkit은cremental adoption이 아니야. 아키텍처가 다르니까, 마이그레이션이 필요해. 
     - Effect에 투자할 준비가 안 된 팀: Foldkit은 Effect를 사용하기 때문에, Effect에 익숙해지지 않으면 사용하기 어렵대. 
     - React 에코시스템이 필요한 프로젝트: Foldkit은 다른 기반을 사용하니까, React 컴포넌트 라이브러리, Next.js, 미들웨어 등을 사용할 수 없어. 
     - 서버사이드 렌더링이 필요한 프로젝트: Foldkit은 클라이언트 사이드 SPA 프레임워크니까, 스태틱 생성이 가능하지만 직접 구현해야 해.
