아니, 아직 아냐. 
아이디어는 동일한데, 사용자 데이터를 공격자로부터 보호하는 것이 목표야. 
암호학적 해시 함수의 정의는 1976년에 화이트필드 디피와 마틴 E. 헬만이 쓴 논문 'New Directions in Cryptography'에서 기초한다. 
무슨 뜻이냐면, 해시는 취약한 데이터를 보호하기 위해 사용된 값을 나타내. 
예를 들어, 웹사이트는 로그인 정보를 평문을 그대로 저장하지 않고, 해시 값을 저장해. 
로그인을 처음 시도할 때, 사용자 비밀번호가 들어오면, 웹사이트는 해시 값을 생성하고 저장해. 
추가적인 로그인 시도에서, 사용자는 계산된 해시 값을 제공하고, 이는 저장된 해시 값과 일치해야 해. 
그러면 로그인이 성공적으로 완료되고, 공격자가 저장된 값을 훔쳐도 비밀번호를 얻을 수 없게 돼. 
근데, 이것은 공격자가 저장된 해시 값을 가져오면, 사용자의 실제 비밀번호가 아니라도 공격에 성공할 수 있는 가능성이 있는 거지. 
아무튼, 이런 공격을 방지하기 위해 비밀번호 해시는 TLS와 같은 암호화 프로토콜과 함께 사용돼. 
그럼, 왜 해시 값을 저장하는 게 비밀번호를 평문으로 저장하는 것보다 안전할까? 
일련의 해시 함수의 주요 속성은 역방향 계산이 불가능하다는 거야. 
즉, 해시 값을 평문으로 되돌릴 수 없어. 
어떻게 보면, 이것은 매우 단순한 개념이지만, 실제로 사용하는 해시 함수는 훨씬 더 복잡해. 
예를 들어, 높은 차수의 다항식과 같은 함수는 전방향으로 계산하기 쉽지만, 역방향으로 계산하기는 매우 어려워. 
또한, 유한체 위에서 계산되는 함수도 존재해. 
이러한 함수를 사용하면 공격자가 비밀번호를 얻는 것을 훨씬 더 어렵게 만들어. 
근데, 이 모든 함수가 항상 안전한 건 아냐. 
예를 들어, 충돌이 발생할 수 있대.
