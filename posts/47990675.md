### 에이전트 하ーネ스란?
에이전트 하ーネ스는 LLM(대형 언어 모델)을 구동하는 반복 루프를 말함. 사용자에게 프롬프트를 보내고, 응답을 받고, 모델이 요청한 툴 호출을 실행하고, 결과를 다시 모델에게 넘겨줌. 그리고 모델이 작업을 끝냈는지 확인하기까지 이 과정을 반복함.
Every production agent has one. The question is where it runs. 
두 가지 답이 있음. 각각의 보안 속성과 오류 모드 및 에이전트가 수행할 수 있는 작업에 대해 차이점이 있음.

### 샌드박스 안에 하ーネ스를 두는 경우
루프는 작업하는 코드와 동일한 컨테이너에 존재함. LLM 호출은 컨테이너 안에서 발생하고, 툴 호출은 로컬에서 실행됨. 하ーネ스가 관리하는 스킬, 메모리 등은 컨테이너의 파일시스템에 파일로 저장됨.
이 방식의 장점은 단순한 실행 모델임. 하나의 컨테이너, 하나의 프로세스 트리, 하나의 파일시스템, 하나의 수명 주기임. 기존 하ーネ스를 그대로 재사용할 수 있음. 스킬과 메모리는 로컬 파일시스템을 가정하므로 변경 없이 사용할 수 있음.

### 샌드박스 밖에 하ーネ스를 두는 경우
루프는 백엔드에서 실행되고, 툴을 호출할 때는 샌드박스에게 API 호출을 함. 샌드박스는 툴을 실행하고 결과를 반환함. 루프는 샌드박스에 들어가지 않음.
이 방식의 장점은 자격 증명 정보가 샌드박스에 없는 것임. LLM API 키, 사용자 토큰, 데이터베이스 접근 정보 등이 하ーネ스에만 존재함. 샌드박스는 에이전트가 작업을 수행하는 데 필요한 환경만 가지고 있음. 그래서 권한 모델을 적용할 필요도 없고, 자격 증명 정보가 유출되는 것도 막을 수 있음.
사용하지 않는 동안 샌드박스를 중지시킬 수 있음. 에이전트가 수행하는 대부분의 작업은 샌드박스가 필요하지 않음.思考, API 호출, 요약, CI 대기 등. 어떤 세션은 샌드박스를 건드리지 않음.
하ーネ스가 샌드박스 밖에 있으면, 샌드박스를 프로비전할 때만 사용하고,闲暇时에는 suspend 할 수 있음. 하ーネ스가 샌드박스 안에 있으면, 이런 것을 할 수 없음.

### 에이전트 하ーネ스 외부에 두는 경우의 해결책
- 지속적인 실행: 에이전트 루프는 긴 실행 함수임.분鐘 단위로 실행되며,.deploy나 scale event, instance 실패에도 생존해야 함. 우리는 Inngest를 사용하여 이를 해결하였음.
- 샌드박스 수명 주기: 에이전트가 명령을 실행하는 동안에만 활성화하고, 그 외의 시간에는_suspend 함. Cold start 문제를 Blaxel을 사용하여 해결하였음.
- 파일시스템: 현대적인 에이전트 하ーネ스는 단순한 bash와 LLM만으로 구성되지 않음. 스킬과 메모리도 존재함. 스킬은 특정 경로의 파일로 저장되고, 메모리는 파일로 저장됨. 그러나 샌드박스가 일시적인 경우, 이러한 파일을 저장하고 읽는 방식이 問題가 됨. 따라서 데이터베이스에 저장하고, 에이전트가 요청할 때 읽어와서 제공함.
- 가상화된インターフェース: 에이전트는 파일을 읽고 쓰는 도구를 사용하여 스킬과 메모리에 접근함. 그러나 실제로는 데이터베이스에 저장된 값임. 이 방식의 문제점은새로운 기능이 추가되었을 때, 이를 처리하기 위한 가상화된 인터페이스를 지원해야 한다는 점임.
