# 깃 히스토리 커맨드, 이제는 더 주목받아야 함
깃으로 작업할 때 여러変更 사항을 동시에 다루는 건 정말 힘듦. 브랜치를 왔다 갔다하면서 커밋을 다시 작성하는 건 정말 정신없고, 혼선이 생길 수도 있음. 요즘은 jj라는 걸 쓰는 사람들도 많지만 나는 이건 쌉별로임. 1.5년 동안 3개월마다 한 번씩 jj를 써봤지만 결국엔 깃으로 돌아가게 되더라고.
그런데 깃 히스토리 커망드가 나왔는데, 이것은 실험적인 커망드라고 하고 깃 2.54, 2.55 버전에서 나옴. 이 커망드는 fixup, reword, split이라는 세 가지 서브 커망드가 있음.
## fixup
fixup은 기존의 커밋을 고칠 수 있는 기능임. 예를 들어 커밋에 에러가 있는 경우, 이 커망드를 쓰면 해당 커밋을 고칠 수 있음. 커망드를 사용하기 전에 `git add`로 스테이징을 해서 커밋을 고친 다음 `git history fixup <커밋명>`으로 커밋을 고친 다음 자동으로 리베이스를 해줌.

예를 들어, B라는 커밋에서 에러가 발생한 경우에 이 커망드를 사용하면 B커밋에 에러가 고쳐지고, C와 D커밋도 자동으로 리베이스됨. 이렇게하면 브랜치에 대한 변경 사항도 자동으로 반영됨.
## reword
reword는 커밋 메시지를 고칠 수 있는 기능임. 예를 들어, 기존의 커밋 메시지가 마음에 들지 않거나, 프로젝트의 설계가 변경된 경우에 이 커망드를 사용하면 커밋 메시지를 고칠 수 있음. `git history reword <커밋명>`을 사용해서 커밋 메시지를 고치면, 이후의 커밋도 자동으로 리베이스되어 변경 사항이 반영됨.
## split
split은 하나의 커밋을 두 개의 커밋으로 나눌 수 있는 기능임. 예를 들어, 하나의 커밋에 여러 개의 변경 사항이 동시에 존재하는 경우에 이 커망드를 사용해서 하나의 커밋을 두 개의 커밋으로 나눌 수 있음. `git history split <커밋명>`을 사용해서 커밋을 나눌 수 있음.
깃 히스토리 커망드는 JJ와 비교했을 때 기능이 약간 약함. JJ는 충돌을 일급 시민으로扱하지만 깃 히스토리 커망드는 아직까지는 자동으로 충돌을 해결할 수 없음. 하지만 깃 히스토리 커망드는 이미 깃에 내장된 기능이므로, 별도로 설치할 필요가 없음. JJ와 비교했을 때 쓰기가 편리하고, 자동으로 리베이스를 해주기 때문에 쓰기가 편리함. 내가 사용하는 깃에 이미 포함되어 있기 때문에 편리하게 사용할 수 있음. 문서가 제대로 작성되어 있어서 이후에 업데이트가 있으면 더 좋아질 거 같은데 기대가 됨.
