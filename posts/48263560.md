> **🔗 원문 보러가기:** [C array types are weird](https://anselmschueler.com/blogposts/2025-c-pointers/)

C 언어를 쓰면서 가장 많이 헷갈리는 건 배열 타입일거임. 
배열 타입은 기본적으로 컨티규어스한 메모리 공간에 있는 값들의 집합을 나타내는 거임. 
하지만 실제로 배열 타입의 값에 접근하는 건 불가능하고, 항상 포인터 타입으로 변환됨. 
예를 들어, `int arr[3] = {10, 20, 30};` 라고 하면, `arr`은 배열 타입이지만, `arr`에 접근하면 항상 `int*` 타입의 포인터로 변환됨. 
이 때문에 배열 인덱싱 연산자 `arr[ix]`는 `*(arr + ix)`와 같은 효과를 냄. 
즉, 배열을 포인터처럼 다루는 건 매우 자연스러운 일이지만, 이 때문에 배열과 포인터를 구분하는 것이 중요해짐. 
특히 함수 시그니처에서 배열 타입을 인자로 전달하면, 배열의 크기 정보는 사라지고 포인터 타입으로만 전달되기 때문에, 
`sizeof` 연산자가 배열의 크기를 반환하지 않음. 
이러한 배열과 포인터의 모호성은 개발자들이 배열과 포인터를 혼용해서 사용하는 것을 방지하기 위해, 
 STRICT한 구분이 필요하고, 배열을 구조체처럼 다루는 방식이 더 직관적이고 쉬울 거임. 
만약 C 언어가 배열과 포인터를 STRICT하게 구분했다면, 
배열을 함수 인자로 전달할 때 실제 배열 값이 복사되어 전달될 거고, 
포인터를 전달하고 싶을 때는 명시적으로 `&arr[0]`과 같이 전달해야 할 거임. 
이러한 방식은 언어를 더 배우기 쉽게 만들뿐만 아니라, 개발자들이 코드를 작성하고理解하기도 더 쉬울 거임. 
하지만, 배열을 값으로 전달하는 방식은 배열 복사가 빈번해질 수 있고, 
그에 따른 성능 저하가 발생할 수 있기 때문에, 개발자는 이러한 부분에 주의해야 할 거임. 
또한, 배열을 구조체처럼 다루려면, 배열을 포인터로 만들 때 `@` 연산자를 사용할 수 있음. 
예를 들어, `*ptr@10`은 `ptr`이 가리키는 값을 시작으로하는 크기 10의 배열을 만들거임. 
이러한 방식은 배열을 더 직관적으로 다룰 수 있게 하지만, 실제로 사용하기 위해서는 더 많은 구현과 연구가 필요할 거임.
