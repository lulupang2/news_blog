# 탈윈드 탈출기
8년 전, 저는 탈윈드를 발견했다는 글을 기쁘게 썼음. 그 당시 저는 CSS 코드를 구조화하는 방법을 몰랐고, Tailwind와는 다소混乱한 상황에서 Tailwind를 선택했어. Tailwind는 많은 작은 사이트를 만들 수 있도록 도와줬지.
지난 주에 몇몇 사이트를 Tailwind에서 벗어나기 위해 semantic HTML + 바니라 CSS로 바꾸는 작업을 함. 그리고 이것은 너무 재밌고 흥미로웠어. 그 때 배운 몇 가지 걸 정리해 보겠어. 전임 프론트엔드 개발자가 아니므로 CSS를 배우는过程은 여러 년 동안斷斷續續한 거였음.

Tailwind는 많은 걸 가르쳐줬어. CSS 구조화에 대해 생각할 때 처음에는 겁을 먹었지. 하지만 CSS 구조화에 대해 말하는 블로그 포스트를 읽기 시작했어. 예를 들어, [A whole cascade of layers](https://www.bram.us/2020/03/18/a-whole-cascade-of-layers/)이나 [How I write CSS in 2024](https://web.archive.org/web/20240224224047/https://bradwoo.github.io/2024/how-i-write-css/) 같은 걸 읽음. 그러면 몇 가지를 깨달을 수 있었어.

- 모든 CSS 코드베이스에는 레이아웃, 폰트, 색상, 공통 컴포넌트 등 다양한 것들이 있음.
- 이 모든 것을 관리하기 위해 시스템이나 가이드라인을 갖는 것이 매우 유용해. 그렇지 않으면 모든 것이 혼란으로 바뀌어 버리겠지.
- Tailwind에는 몇 가지 시스템이 있음. 그중에 몇 가지는 이미 알고 있었어. 좋아하는 시스템을 모방하는 건 어떤가?

예를 들어, Tailwind에는 다음과 같은 시스템이 있음:
- 리셋 스타일시트
- 색상 팔레트
- 폰트 스케일

이러한 시스템에 대해 더 자세히 설명해 볼게.
1. **리셋**
Tailwind의 `preflight styles`를 복사했어. Tailwind의 CSS 리셋에 익숙해졌어. 예를 들어, Tailwind는 모든 요소에 `box-sizing: border-box`를 설정해 줌.
```css
* {
  box-sizing: border-box;
}
```
이런 리셋 스타일시트를 쓰지 않으면 많은 것을 다시 배워야 할 거야.

2. **컴포넌트**
이 다음 phần은 대부분의 CSS임. 컴포넌트별로 CSS를 구성하는 생각이야. Vue나 React 컴포넌트와는 다르지만, 비슷한 아이디어야. 이 아이디어는 다음과 같아:
- 각 컴포넌트에는 고유한 클래스가 있어야 함.
- 한 컴포넌트의 CSS는 다른 컴포넌트의 CSS를 덮어써서는 안 돼.
- 각 컴포넌트에는自己的 CSS 파일이 있어야 해.
- 그래서 한 컴포넌트의 CSS를 편집할 때, 다른 컴포넌트의 CSS를意外로 깨뜨리지 않을 거야.

예를 들어, 다음 HTML은 `.zine` 컴포넌트일 수 있어:
```html
<figure class='zine horizontal'>
  <img src='whatever.jpg'>
</figure>
```
그리고 다음 CSS는 `.zine` 컴포넌트의 스타일을 정의해 줌:
```css
.zine {
  /* ... */
  &.horizontal {
    /* ... */
  }
  &.vertical {
    /* ... */
  }
  &:hover {
    /* ... */
  }
}
```
 아직 웹 컴포넌트나 `@scope` 같은 것으로 컴포넌트 간의 충돌을 방지하는 것을 구현하지 않았지만, 단순히 규칙을 정하고 이를 지키려고 노력하는 것만으로도 큰 발전이라고 생각해.

다음으로, 사이트 전체에서 일관성을 유지하기 위한 규칙을 정의해 볼게.
3. **색상**
`colours.css` 파일에는 다음과 같은 변수들이 있어:
```css
:root {
  --pink: #fea0c2;
  --pink-light: #F9B9B9;
  --red: #f91a55;
  --orange: rgb(222, 117, 31);
  /* ... */
}
```
색상은 thật難，所以 이 번역에서는 색상을 다시 보지 않기로 했어. 색상 파일의 유일한 규칙은 사이트에서 사용되는 모든 색상이 이 파일에 나열된다는 거야.

4. **폰트 크기**
Tailwind를 쓸 때, 폰트 크기를 설정할 때는 просто `text-lg`라고 쓰면 됐어. 하지만 Tailwind를 떠나면서, 다음과 같은 변수들을 정의했어:
```css
--size-xs: 0.75rem;
--line-height-xs: 1rem;
--size-sm: 0.875rem;
--line-height-sm: 1.25rem;
```
이제 폰트 크기를 설정할 때는 다음과 같이 할 수 있어:
```css
h3 {
  font-size: var(--size-lg);
  line-height: var(--line-height-lg);
}
```
5. **유틸리티**
按钮이나 기타 여러 컴포넌트에서 사용되는 요소들을 유틸리티라고 부를게. Tailwind의 유틸리티 클래스 중 몇 가지는 복사했어. 이 섹션은 상대적으로 작고, 여기서 변경을 melakukan 것은 신중해야 해.

6. **베이스 스타일**
전체 사이트에 걸쳐 적용되는 스타일을 베이스 스타일이라고 할게. 이 섹션은 매우 작아야 해. 현재 두 가지 스타일만이 만족스러워. `<section>` 관련 스타일은 나중에 변경될 수도 있어:
```css
/* 각 <section>에 950px의 열을 생성 */
section {
  --inner-width: 950px;
  padding: 3rem max(1rem, (100% - var(--inner-width))/2);
}
a {
  color: var(--orange);
}
```
베이스 스타일은 보통 아래에서 위로 작업하는 게 좋을 것 같아. 먼저 거의 아무 스타일도 없이 시작하고, 컴포넌트에서 공통적인 스타일을 발견할 때마다 컴포넌트에서 베이스 스타일로 옮길 수 있어.

7. **스페이싱**
まだ padding 과 margin을 관리하는 접근 방법을 완전히 결정하지 못했어. Tailwind를 쓰던 때보다 더 원칙적으로 관리하고 싶어. 현재 OUTER 레이아웃 컴포넌트가 spacing을负责하는 방향으로 작업중이야. 예를 들어, `<section>`에 여러 자식 요소가 있고, 이들 사이에 공백을 주고 싶은 경우 다음 코드를 사용할 수 있어:
```css
section > *+* {
  margin-top: 1rem;
}
```
8. **响应式 설계**
Tailwind를 사용할 때, 많은 미디어 쿼리를 사용해야 했어. 하지만 이제는 CSS Grid 레이아웃을 사용해서 더 유연한 설계를 할 수 있어. 예를 들어, 다음 코드는 자동으로 큰 화면에서 2 칼럼, 작은 화면에서 1 칼럼을 생성해 줌:
```css
display: grid;
grid-template-columns: repeat(auto-fit, minmax(min(100%, 400px), max-content));
justify-content: center;
```
9. **빌드 시스템**
개발 중에는 빌드 시스템이 필요 없지만, 프로덕션으로 가기 전에 esbuild를 사용해 CSS 파일을 번들할 수 있어.
```bash
esbuild style.css --bundle --loader:.svg=dataurl --loader:.woff2=file --outfile=/tmp/out.css
```
Tailwind를 쓰던 때와는 다르게, 빌드 시스템이 필요 없어서 좋았어.

왜 Tailwind를 포기했을까?
몇 가지 이유가 있었어. Tailwind는 2018년 이후로 빌드 시스템에 많이 의존하게 됐고, 최신 버전을 사용하려면 빌드 시스템이 필요해졌어. 그래서 Tailwind v2를 몇 년 동안 사용했어. 빌드 시스템을 사용하지 않고 Tailwind를 사용하는 건ちょっと 어리석은 짓 같았어. CSS를 더 잘 알게 된 후에, Tailwind의 제약에 대한 불만이 커졌어. 예를 들어, Tailwind에서 할 수 없는 Weird한 일을 하고 싶었어. 또한 Tailwind와 바니라 CSS를 混在해서 사용하는 사이트를 관리하는 것이 힘들어졌어.

이번에 배운 CSS 기능 중에서 아직 사용하지 않은 기능이 몇 가졌어. 예를 들어, `@layer`, `@scope`, container queries, subgrid 등이 있잖아.

지난 3년 전, [Tailwind and the Femininity of CSS](https://www.swyx.io/talks/tailwind-femininity-cs/)라는 포스트를 읽었었어. 이 포스트는 CSS를簡單하게 만들기 위한 Tailwind의 접근 방법을 비판하는 거야. 그 후로, CSS를爱하고尊敬하는 개발자가 되고 싶다고 생각했어. 그래서 Tailwind를 떠나기로 quyết심했어.
