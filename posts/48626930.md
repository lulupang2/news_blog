> **🔗 원문 보러가기:** [Codex logging bug may write TBs to local SSDs](https://github.com/openai/codex/issues/28224)

### 문제 상황
코덱스 sqlite 로그가 엄청나게 많이 쌓여서 ssd의 수명이 급격히 줄어드는 버그가 있대 ㅋㅋㅋ
원인은 sqlite 로그가 너무 많이 기록되는데, 이게 640tb까지도 쌓일 수 있단다. 
这样 하면 1tb ssd는 1년 안에 수명이 다할 수도 있대. 
근본 원인은 코덱스 sqlite 로그가 TRACE 레벨의 로그를 기본적으로 사용하기 때문에 너무 많이 기록된단다. 
이거 때문에 dependency(internal) 로그와 raw protocol payload 도 많이 기록된대. 
### 해결 방법
해결 방법은 몇 가지가 있는데,
1. sqlite 로그의 디폴트 레벨을 TRACE로부터 INFO나 DEBUG 레벨로 바꾸는 거임. 
2. 로그를 너무 많이 기록하는 target(log, hyper_util, tokio-tungstenite internals, inotify spam, low-level OpenTelemetry SDK logs)을 필터링 해버리는 거고, 
3. raw websocket/sse payload를 기록하지 말고, 대신에 요약 정보만 기록해버리면 됨. 
4. sqlite 로그 용량 제한을 걸어버리면 됨. 
이런 방법으로 로그 파일의 크기를 관리할 수 있겠네. 
개빡치는 점은 이런 문제가 이전에도 있었으면서 아직도解决되지 않고 있다는 거임. 
코덱스 개발팀이 좀 더 주의 깊게 관리해주면 좋겠네.
