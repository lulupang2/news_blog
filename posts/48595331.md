### Well-Known URI 쓰는 방법
웹開発자들은 새로운 Well-Known URI를 정의할 때, 여러 가지 고려해야 할 점이 있음. 
마크 노팅엄이라는 웹 개발자曰, Well-Known URI는 사이트의 특정 부분을 효율적으로 탐색할 때 사용되며, 로봇 텍스트 파일은 그 예시 중 하나임.
하지만, 일부 개발자들은 Well-Known URI를 잘못 사용하는 경우가 많단다. 
예를 들어, 일부 개발자들은 Well-Known URI를 단순히 URL 단축기처럼 사용하거나, 해당 프로토콜이 실제로 요구하지 않는 경우에 사용하곤 함.
### Well-Known URI의 올바른 사용법
Well-Known URI는 클라이언트가 사이트를 이미 알고 있고, 사이트 전체에 대한 정보를 효율적으로 검색해야 하는 경우에 사용됨.
예를 들어, 로봇 텍스트 파일은 사이트의 접근 정책을 나타내며, 이를 통해 크롤러가 사이트의 정책을 쉽게 학습할 수 있음.
하지만, Well-Known URI는 항상 정책을 포함하지는 않음. 
클라이언트가 사이트를 이미 알고 있고, 사이트 전체에 대한 정보를 효율적으로 学習해야 하는 모든 메커니즘은 Well-Known URI로 등록할 수 있는 후보임.
예를 들어, 패스워드 변경 Well-Known URI는 클라이언트가 사이트의 패스워드를 변경할 수 있음.
### 잘못된 사용법
Well-Known URI는 일부 프로토콜에서 실제로 필요한 경우가 있음에도, 잘못 사용되는 경우가 많단다.
일부 개발자들은 Well-Known URI를 등록하면 프로토콜의 채택이나 합법성이증가한다고 생각하는 경우가 많음.
하지만, Well-Known URI는 단지 특정 문제를 해결하는 도구일 뿐임.
만약 프로토콜이 Well-Known URI가 해결하는 문제를 가지고 있지 않다면, 등록은 새로운 문제를 생성할 뿐임.
### 일반적인 함정과 트레이드오프
Well-Known URI는 올바른 도구지만, 일부 가정은 항상 참이 아닐 수 있음.
Well-Known URI를 정의할 때, 다음 사항을 고려해야 함:
*   발견 메커니즘: Well-Known URI는 발견 메커니즘으로 사용되곤 하는데, 이는 클라이언트가 사이트를 이미 알고 있다고 가정하기 때문임.
    하지만, 현실은 항상 그런 것은 아니며, 사용자의 현재 상호작용 범위와 발견하는 범위가 다를 수 있음.
*   콘텐츠 메타데이터: 일부 프로토콜은 Well-Known URI를 콘텐츠 메타데이터를 학습하는 방법으로 사용하곤 함.
    하지만, 이러한 패턴은 항상 동작하지는 않으며, 사이트가 여러 발행자를 나타내는 경우 복잡한 인프라를 요구할 수 있음.
