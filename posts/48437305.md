### 개빡치는 시작
나는 AI랑 코드 짜看到 early버젼에서 내가 AI한테 너무 많이 맡긴다는 생각이 들어서 코드를 신뢰하지 않게 되었어. 
하지만trust를 회복하기 위해 내가할수있는 모든 의심을 자동화 해봤는데, 결과적으로 코드를 훨씬 더 신뢰하게 된거 같음. 

일단 코드의 구현을 비판하고, 그것을 반복하는 것이 의심을 하는 것의 의미를 담고 있어. 
만약你가 AI를 사용해 코드나 문서를 작성하고 있다면, 이 글을 읽는 것 만으로도 도움이 될 거야. 
나는 서브 에이전트들을 많이 사용하는데, 이들은 프로세스 전체의 핵심을 차지하고 있어. 
특정한 방식으로_audit을 통해 표준 Claude에서 다루지 못하는_surface를 다뤄. 
이 모든 것의 핵심은 여러 측면에서 자동화된 의심 과 front-loading of scrutiny이야. 
더 많은 parallax coverage가 있는 AI 개발이 더 좋은 선택일 거야. 
여기서 두 개의 눈이深度를 줄 수 있는 것 처럼 다른 관점에서 다른 결함을 잡을 수 있어. 

이 Developement 과정은 3단계로 나눠졌어. 
Phase 1에는 Design이 있고, 그 다음은 개발 그리고 마지막에는_ship이 있어. 

### Phase 1 - Design
일단 시작은 아이디어나 빌드하고 싶은 기능과 specification으로 시작해. 
디자인은 좋은 개발 관습 처럼 spec, PRD, plan 등으로 시작하는 게 좋아. 
나는 Claude에게 spec을 작성하고, 2~5분 정도 파일을 훑어보며 핵심 구현 측면이 잘 담겨 있는지 확인 해. 
그리고 pre-implementation workflow를 시작해. 
slash command를 이용해서 3개 agents가 돌아가면서 의심을 해. 
Pre-Implementation Architect, Documentation Validator, Assumption Excavator가 그 이름이야. 
이 에이전트들은 디자인 품질을 검증하고, 범위를 평가하고, 문서화된 내용을 점검해. 
모든相關 결과를 spec에 담아. 

예를 들어 Assumption Excavator는 `
executionStatsSchema in registry-sdk returns {totalCount, recentCount, windowMinutes}. 
Spec assumes {avgScore, medianDurationMs, passRate, lastRunDate, lastRunScore}. 
Entire history section unbuildable without new API endpoint`라고 알려줬어. 
Pre-Implementation Architect는 `
HarnessProfile embeds mcp.read/merge/remove/write methods alongside path config. 
Consider extracting McpConfigStrategy to separate concerns. 
Each harness file will grow to 80~120 lines otherwise.`라고 알려줬어. 

모든 결과를 spec에 담은 다음, 다음 에이전트들을 이용해서 gap을 분석하고 implied completeness를 확인하고 ambiguity를 mapping 해. 
 Gap Analyst, Implied Completeness Detector, Ambiguity Mapper가 그 이름이야. 
이 에이전트들은 시스템에서 누락된 부분을 찾아내고, 그것을 spec에 추가해. 
예를 들어 Gap Analyst는 `
McpConfigStrategy defines read/merge/write but does not specify behavior for malformed input, permission denied, partial write failure, or file locking. 
Destructive operation on user config files across 4 harnesses in 3 formats.`라고 알려줬어. 
Implied Completeness Detector는 `
Manifest records version at root but installation state per-harness. 
When v0.3.0 user (Claude Code) runs v0.4.0 with --harness opencode, behavior undefined. 
Per-harness versioning or upgrade reconciliation not addressed.`라고 알려줬어. 

 πραctical한 관점에서 보면_small scope라면 pre-implementation만으로 충분하지만, medium scope라면 pre-implementation와 gap, implied, ambiguity를 함께 사용해. 
large scope라면 전체 단계를 여러 번 반복해. 

이 모든 절차를 끝내면 15~60분 정도를 투자해서 spec을 읽어. 
그럼 spec이 개발에 sẵn sàng하다고 생각되면, 나는 Claude에게 companion checklist를 생성해. 
이 체크리스트는 spec과 함께 따로 파일을 생성해. 
개발 중간에 떠나거나 중지할 필요가 있을 때 도움이 될거야. 

### Phase 2 - Development
개발을 시작하면 Claude가 spec과 체크리스트를 불러와서 시작해. 
만약 개발을 부분적으로 완료했으면, 나는 Claude에게 explore를 시키거나 Chain Tracer나 Deep Explore subagent를 사용해서 전체적인 그림을 확인해. 
개발过程에서 특징적인 점은 나는 에이전트를 이용해서 실제로 코드를 작성하지 않아. 
트러스트의 문제 때문이야. 
에이전트를 이용해서 코드를 작성하다가 더 많은 문제를 일으키는 경우가 많았어서, 일시적으로 그것을 중단했어. 
클라우드 테르미널 에이전트가 Bulk updates를 위해 에이전트를 생성할 수도 있지만, 나는 하나의 클라우드 코드 터미널 인스턴스를 이용해서 프로젝트를 구축하는 것을 선호해. 

### Phase 3 - Wrap-up and Ship
모든 단계를 끝내면, 나는 Post-Implementation workflow를 다시 시작해. 
이번에는 Code Validator, Type Safety Validator, Test Architect, Code Optimizer, Public Interface Validator, Security Analyst를 이용해. 
이 에이전트들은 코드베이스를审核하고 findings를 제공해. 
코드의 품질, 보안, 중복, 성능, 문서화 등에 대한 findings를 주는데, 대개 15~35개의 findings를 주고, 처음 15~20개는 critical 또는 high severity로 표시해. 
이 findings를 해결하고, 다시 Post-implementation workflow를 실행해. 
이렇게 여러 번 반복해서 모든 issues를 해결해. 
예를 들어 Code Validator는 `
Every other execution method calls trackIfEnabled() after completion. 
startPipeline() returns PipelineHandle directly without tracking. 
Async pipeline users get no tracking data.`라고 알려줬어. 
Security Analyst는 `
PreflightError includes shellQuote-expanded target path verbatim. 
Error messages containing resolved filesystem paths may propagate to tracking API and dashboard.`라고 알려줬어. 

Ship 단계에서는 최종적인 iterative process를 수행해. 
이 단계에서는 Code Validator, Type Safety Validator, Test Architect, Code Auditor, Public Interface Validator, Security Analyst, Anxiety Reader, API Contract Validator를 이용해. 
이 에이전트들은 API의 계약을 확인하고, 런타임 일관성, 시스템의 릴리즈姿态를 점검해. 
이 workflow가 실행되면, 릴리즈 준비가 되었는지 확인할 수 있어. 
complexity에 따라 여러 번 iteration이 필요할 수 있어. 
예를 들어 Anxiety Reader는 `
Promise.allSettled fires all agents simultaneously with no concurrency limit, risking resource exhaustion and API rate limits.`라고 알려줬어. 
Code Auditor는 `
File I/O errors in writeReportFiles caught by handleCoreError which gives SDK-specific hints instead of filesystem-specific messaging.`라고 알려줬어. 

결론적으로 이 개발 프로세스는 자동화된 의심과 front-loading of scrutiny에 관한 이야기야. 
이 모든 것을 가능하게 하는 것은 에이전트들이야. 
나는 에이전트들을 이용해서 코드를 개발하고, 그 코드를 검증하고, 릴리즈 준비를 해. 
그 과정에서 문제점을 찾아내고, 그것을 해결해. 
그리고 릴리즈를 준비해. 
이 모든 것을 하기 위해 에이전트들을 이용하고, 자동화된 의심을 이용해. 
그것이 내가 사용하는 개발 프로세스야. 
그리고 그것이 왜 필요한 건지, 그것이 어떻게 도움이 되는 건지에 관한이야기야. 
내가 작성한 코드가 AI 시스템의 도움으로 작성되었어도, 그것이 검증되고, 검사되고, 릴리즈 준비가 되었는지 확인하기 위해, 나는이 프로세스를 사용해. 
그리고 그것이 효과가 있음을確認했다
