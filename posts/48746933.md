### 요즘 컴퍼일러는 알아서 잘 최적화해 준다고?
   
   요즘 C++ 프로그래μερ들은 오스모시스 통해 성능에 관련된 많은 지식들을 흡수할 수 있음.
   예를 들어 빠른 역 平方근, XOR 스왑, 手-언-오-루-드 루프, Duff's 디바이스, 예외는 느리다, 가상 호출은 느리다, 나눗셈은 느리다, 룩업 테이블은 수학 계산 보다 빠르다 등이 있음.
   
   하지만 하드웨어, 소프트웨어, 컴파일러가 발전하면서 이런 지식은 대부분 과거의 유물이 되어버림.
   
   옛날 DEC Alpha, 펜티엄 프로와 비교하면 요즘 Zen 5 x86-64 코어는 별개의 것임.
   예를 들어 브랜치 예측기는 옛날 워크스테이션 보다 더 많은 트랜지스터가 사용됨.
   
   LLVM의 Clang과 GNU의 GCC는 나이브 입력으로부터 최적의 코드를 생성해 낼 수 있을 정도로 발전했음.
   
   예를 들어 두 컴파일러 모두 팝카운트를 위한 최적화 파스를 가지고 있어 코드 최적화를 할 수 있음.
   " Clever " 코드는 오히려 최적화에 방해가 되고, 벡터화, 인라이닝, 대상-특정 최적화를 할 수 없게 만드는 문제가 있음.
   
   ### 예시 1: 빠른 역平方근
   
   Quake III에서 사용했던 빠른 역 平方근 알고리즘을 보자.
   float Q_rsqrt( float number ) { long i; float x2, y; const float threehalfs = 1.5F; x2 = number * 0.5F; y = number; i = * ( long * ) &y; i = 0x5f3759df - ( i >> 1 ); y = * ( float * ) &i; y = y * ( threehalfs - ( x2 * y * y ) ); return y; }
   
   요즘 컴퓨터에서는 rsqrtss, rsqrtps 같은 명령어가 존재해서 역 平方근을 빠르게 계산할 수 있음.
   
   C++에서는 std::bit_cast 함수를 사용해서 비트 연산을 수행할 수 있음.
   ```cpp
   constexpr float Q_rsqrt(float number) noexcept { static_assert(sizeof(float) == sizeof(std::uint32_t)); auto i = std::bit_cast<std::uint32_t>(number); auto magic = 0x5f3759dfu - (i >> 1); auto y = std::bit_cast<float>(magic); return y * (1.5f - (number * 0.5f * y * y)); }
   ```
   
   벤치마크 결과, 두 함수의 성능 차이가 거의 없음.
   
   요즘 코드를 작성할 때는 컴파일러가 알아서 최적화해 준다고 믿고 코드를 작성하는 것이 좋음.
   
   ### TAKE AWAY
   
   *   컴파일러가 알아서 최적화해 줄 거라고 믿고 코드를 작성하자.
   *   C++20에서 추가된 std::popcount 함수를 사용하면 비트 연산을 수행할 수 있음.
   *   row-pointer 보다 multiplication을 사용하는게 성능상 좋음.
   *   const&를 쓸 때는 함수가 argument를观察만 할 때 사용하고, wrapper code에서는 const&를 사용하지 말자.
