### 홈스케일: 플래닛스케일을 집에서 만들기
플래니트스케일 따라해 보겠다는 이야기를 최근에 듣긴 했지만, 내가 데이터베이스 툴ズ 회사에서 일할 당시 몇가지 데이터베이스 클로닝 툴을 만들었었음. 그 툴들이 플래닛스케일만큼 성공은 못했지만,还是 그 툴들은 쓰임새가 있었지. 근데 그 툴의 핵심 아이디어는 간단했어. 스토리지 레이어를 컴퓨팅 레이어랑 분리하고, 데이터ベース 파일 복제에 있어서 가장 좋은 스토리지 기술을 쓴다는 거였어.
 
 근데 내가 예전에 F1 팀하고 인터뷰한 적 있는데, 러스트와 셉을 다룰 수 있는 개발자를 찾고 있었지. 셉 얘기가 나와서 내가 예전 셉 관련 프로젝트에서 실패한 경험이 떠오르면서, 친구랑 얘기하던 중에 홈스케일, 플래닛스케일을 집에서 만들어 보기로 했다는 얘길 했었어. 그 친구가 그 말을 믿지 않았지만, 내가 실제로 만들어 보기로 했음.
 
 ### 홈스케일 깃허브 저장소
홈스케일 깃허브 저장소: https://github.com/homescale-dev/homescale
홈스케일은 데이터베이스 인스턴스를 만드는 툴이고, 특징은 불변 스냅샷에서Writable한 인스턴스를 생성할 수 있고, 포인트 인 타임 브랜치를 생성할 수 있다는 점임.
 
### CLI 사용 예시
docker 이미지를参考해서 데이터베이스 상태를 표현하기로 했어.
*   데이터베이스 이미지: 불변한 시작점
*   데이터베이스 컨테이너: 이미지에서 생성된 쓰기 가능한 클론
*   브랜치: 컨테이너에서 만들어진 다른 컨테이너
 CLI 사용 예시는 다음과 같아.
 ```
homescale image create --engine postgres postgres-base
homescale container create --image postgres-base dev-db
homescale container connect dev-db
homescale branch create --container dev-db feature-login
homescale container connect feature-login
```
### engine
데이터베이스 엔진 아에 구애받지 않고 쓰일 수 있는 모델을 지향했어. 포스트그레Sql부터 시작해서 다른 엔진도 지원 가능하도록 설계했지. 엔진 별로 적응해야 하는 내용은 어댑터 뒤로 숨겨서, 어댑터는 이미지 초기화, 데이터베이스 프로세스 시작, 커넥션 정보 노출, 스냅샷 준비 등을 담당하게 했어. 홈스ケ일은 어댑터 周辺을 담당하게 했음. 볼륨, 불변 상태, 쓰기 가능한 복제본, 워크로드, 계보 등의 라이프사이클을 관리하는 거임.
 
### 브랜치
dev-db 와 feature-login 두 컨테이너가 서로 어떤 관계인지 보자면, feature-login 는 dev-db 에서 생성된거고, dev-db 도 postgres-base 에서 생성된거임. 이처럼 서로 컨테이너 간의 관계를 flowchart 로 그리면 다음과 같아.
 ```mermaid
flowchart LR
    Image[/postgres-base<br/>image/] -->|clone|> Dev[dev-db<br/>writable]
    Dev -->|snapshot|> State[/read-only<br/>state/]
    State -->|clone|> Feature[feature-login<br/>writable]
```
 이 모델에서는 이미지를 불변으로 두기 때문에, 이미지에서(dev-db) 바로 컨테이너를 생성할 수록 되고, 컨테이너에서(feature-login) 또 다른 컨테이너를 생성하려면, 중간에 불변 상태(Intermediate State)를 생성해야 해.
 
 ### Ceph 
 Ceph 는 object, file, block 스토리지 모두 지원하는 분산 스토리지 시스템인데, 여기에 대한 자세한 설명은 https://docs.ceph.com/en/pacific/start/intro/ 에서 더 확인하길 바래.
 ```mermaid
flowchart TB
    Database[Database process] -->|file reads and writes|> Filesystem[Filesystem]
    Filesystem -->|block I/O|> Device[Block device]
    Device --> Storage[(Persistent storage)]
```
 Ceph 는 RBD(image)를통해서 블록 디바이스처럼 사용할 수 있는데, RBD 는RADOS(object store) 위에 있는 블록 디바이스를 말해.
 
### RADOS
RADOS 는 object store 라고 부를 수 있지. object 에대한 설명은 https://docs.ceph.com/en/pacific/rados/theory/#objects 에서 더 확인할 수 있어.
```mermaid
flowchart TB
    RBD[RBD image] -->|split into|> Objects[RADOS objects]
    Objects -->|hash to|> PG[Placement group]
    PG -->|maps to|> OSDs[Acting OSD set]
    OSDs -->|persist on|> Disks[(Storage devices)]
```
 OSDs 는 RADOS 의 핵심コンポーネント라고 불릴 수 있지. RADOS는 object 를 OSD 에 저장하고 불러오기 위한 매커니즘을 제공해주고, OSDs 는 RADOS对象를 실제 디스크에 저장하는 역할을 해.
 
 ### COW
copy-on-write 는 내가 컨테이너 생성 시 쓰기 가능한 클론을 만들고, 원본의 데이터를 그대로 사용하는 방법인데, 이는 데이터가 실제로 변경될 때만 추가적인 스토리지를 사용하는 방식이거든. 
```mermaid
flowchart LR
    Image[/postgres-base<br/>image/] -->|clone|> Dev[dev-db<br/>writable]
    Dev -->|snapshot|> State[/read-only<br/>state/]
    State -->|clone|> Feature[feature-login<br/>writable]
```
이런식으로 클론을 생성한 후에, 원본에서 변경된 것을 클론이 그대로 사용하고, 클론에서 변경된 것을 원본이 사용하지 않도록 하는 거임. 그러다가 클론에서 원본의 데이터를 사용하다가, 클론에서 데이터를 변경하면, 클론이 데이터를 복사해서 사용하는 방식이거든. 그럼 이렇게되면 원본의 데이터와 클론의 데이터는 서로 다른 데이터가 되지.
 
  ### 요약
 요약하자면 홈스케일은 데이터베이스에 대한 불변 상태와 쓰기 가능한 복제본을 생성하는 툴이며, Ceph 의 RBD 를 사용해서 이것을 구현하고 있지. RADOS 는 Ceph 의 핵심コンポーネント로, object 를 저장하고 관리하는 역할을 하고, OSDs 는 RADOS对象를 실제 디스크에 저장하는 역할을 해. copy-on-write 는 내가 컨테이너 생성 시 쓰기 가능한 클론을 만들고, 원본의 데이터를 그대로 사용하는 방법인데, 이는 데이터가 실제로 변경될 때만 추가적인 스토리지를 사용하는 방식이거든.
