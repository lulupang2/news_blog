### 파싱은 검증이 아님

 최근에 Alexis King님의 'Parse, don't validate'라는 글을 다시 읽게 되었다. 이 글은 2019년에 작성되었지만, 여전히 타입스크립트 개발자에게 중요한 정보를 제공한다.

 검증과 파싱의 차이점은 무엇일까? 검증은 단순히 입력 데이터가 올바른지 확인하는 것이고, 파싱은 입력 데이터를 더 정확한 타입으로 변환하는 것이다. 파싱을 통해 얻은 정보는 코드의 다른 부분에서 사용할 수 있기 때문에, 코드를 더 안전하고 유지하기 쉬운 방식으로 작성할 수 있다.

### 타입스크립트에서 파싱하기

타입스크립트는 구조적 타이핑 언어이다. 즉, 두 개의 타입이 같은 모양을 가지면, 같은 타입으로 간주된다. 하지만, 이 방식은 파싱을 어렵게 만든다. 왜냐하면, 타입스크립트는 새로운 타입을 만들기 어렵기 때문이다.

이를 해결하기 위한 방법은 브랜딩(branding)이다. 브랜딩은 새로운 타입을 만드는 방법으로, 기존 타입에 추가적인 정보를 더하는 방식이다. 예를 들어, `string` 타입에 `__brand`라는 속성을 추가하여 `Email` 타입을 만들 수 있다.

```typescript
type Email = string & { readonly __brand: 'Email' };
```

이 방식은 타입스크립트의 구조적 타이핑을 이용하여 새로운 타입을 만들 수 있지만, 약간의 번거로움이 따른다.

### 파싱 함수를 작성하기

파싱 함수는 입력 데이터를 더 정확한 타입으로 변환하는 함수이다. 예를 들어, 이메일 주소를 파싱하는 함수를 작성할 수 있다.

```typescript
function parseEmail(raw: string): Parsed<Email> {
  if (!raw.includes('@')) {
    return { kind: 'err', error: { kind: 'ParseError', message: 'missing @' } };
  }
  return { kind: 'ok', value: raw as Email };
}
```

이 함수는 입력 데이터를 검증하고, 더 정확한 타입으로 변환한다.

### 파싱의 이점

파싱을 통해 얻을 수 있는 이점은 다음과 같다.

*   코드를 더 안전하고 유지하기 쉬운 방식으로 작성할 수 있다.
*   입력 데이터를 더 정확한 타입으로 변환할 수 있다.
*   코드의 다른 부분에서 파싱된 정보를 사용할 수 있다.

### 결론

파싱은 검증이 아니다. 파싱을 통해 더 안전하고 유지하기 쉬운 코드를 작성할 수 있다. 타입스크립트에서 파싱을 하기 위해서는 브랜딩을 사용할 수 있다. 파싱 함수를 작성하여 입력 데이터를 더 정확한 타입으로 변환할 수 있다. 파싱의 이점은 코드를 더 안전하고 유지하기 쉬운 방식으로 작성할 수 있고, 입력 데이터를 더 정확한 타입으로 변환할 수 있다.
