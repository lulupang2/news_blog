나는 이맥스랑 i3를 통합해봤다 ㅋㅋㅋㅋ
   
   일단 타일링 윈도우 매니저는 정말 좋다. 
   그리고 초フレックス テキスト エ디터도 좋은 것 같음. 
   하지만 exwm를 사용하다가 그래픽 윈도우를 많이 사용하게 되는데 이때문에 문제가 생김. 
   그래도 이맥스는 좋다. 
   예를 들어 내 기계에서 밝기와 어둡기 모드를 전환해 줌 ㅋㅋㅋ. 
   그래서 [제목 없는 게시물](https://news.ycombinator.com/item?id=123456) 같은 글을 보고 
   이맥스와 i3의 공통 키바인딩을 설정해 Boylet. 
   
   처음에는 xdotool과 emacsclient를 사용하는 스크립트를 작성해 봤으나 
   이방법은 느렸음 ㅋㅋㅋ. 
   그래서 i3를 패치해 봄. 
   objetivo는 현재 포커스된 윈도우가 이맥스인지 확인하고, 
   맞으면 키 이벤트를 전달하는 것임. 
   
   이는 이전에도 요청된 기능이지만 i3 개발자가 out of scope라고 판단함. 
   그래서 더 많은 기능을 추가하는 것보다 이 기능만 패치해 보았음. 
   만약 이맥스가 이 기능을 사용하지 않으려면 i3-msg를 사용해 역할을 바꿀 수 있음. 
   
   i3의 Relevant 코드는 [i3wm(i3 4.25.1)](https://github.com/i3/i3/blob/4.25.1/src/bindings.c)임. 
   
   172 struct Binding_Keycode *binding_keycode; 
   173 TAILQ_FOREACH(binding_keycode, &(bind->keycodes_head), keycodes) { 
   174    const int keycode = binding_keycode->keycode; 
   175    const int mods = (binding_keycode->modifiers & 0xFFFF); 
   176    DLOG("Binding %p Grabbing keycode %d with mods %d\n", bind, keycode, mods); 
   177    xcb_grab_key(conn, 0, root, mods, keycode, XCB_GRAB_MODE_ASYNC, 
   178                  XCB_GRAB_MODE_ASYNC); 
   179 } 
   
   이는 i3가 루트 윈도우에서 키 이벤트를攔截하는 코드임. 
   
   handle_key_press()는 キー 이벤트를 받아 키 바인딩을 확인하고, 
   키 바인딩이 있으면 해당 명령을 실행하는 함수임. 
   
   이 함수에서 キー 이벤트를 직접 전달할 수 있지만, 
   이때문에 i3가 키 이벤트를攔截해 버리지는 않도록 해야 함. 
   
  Binding 구조체를 변경해 passthrough 필드를 추가함. 
   
   이 필드는 키 이벤트를 직접 전달할 윈도우 클래스를 나타냄. 
   
   handle_key_press()를 변경해 키 이벤트를 전달할지 말지 결정함. 
   
   마지막으로 이맥스에서 키 이벤트를 받을 수 있도록 elisp를 작성함. 
   
   이는 [제목 없는 게시물](https://news.ycombinator.com/item?id=123456)에서 가져온 코드임. 
   
   키 이벤트를 받으면 i3-msg를 사용해 i3에게 명령을 전달함. 
   
   예를 들어 윈도우를 이동할 때 이맥스에서 직접 이동할 수 있는지 확인하고, 
   불가하면 i3에게 이동을 요청함.
