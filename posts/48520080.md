# 소개
베링거 DDX3216 라는 믹서기 기계에 DIY해서 만든 x86 바이오스를 때려박아 보니까竟然 성공함 ㅋㅋㅋㅋ

## 내 첫 컴퓨터
94년 처음 کامپی를 샀는데 당시엔 인텔 i486 DX2-66 프로세서에 4MB RAM과 512MB 하드 디스크 달려 있던 거였음. 쓰던 운영체제는 IBM의 OS/2와 마이크로소프트의 윈도우 3.11이었지. 첫 컴퓨타 샀을 때부터 4년 동안 계속 업그레이드 하던 거였는데, 램은 16MB까지 올려보고 CD롬 드라이브와 사운드 블래스터 카드도 달아봤었지. 그래서 컴퓨터 업그레이드하고, 소프트웨어 설치하고, BASIC 언어로 프로그래밍도 직접 해봤었어. 하지만 부트 프로세스나 MS-DOS의 자세한 내용은 잘 몰랐었는데, 2026년 yani 32년 후에야 DDX3216이 진짜 386 프로세서를 쓰고 있다는 사실을 알게 되고, 이 기계에 운영체제를 올려 보겠다는 생각이突然 들었어.

## 목표
x86 시스템이怎么 부트되는지, DOS가 어떻게 작동하는지, 쉘에 어떻게 들어가는지 등에 대해 자세히 공부해 보기로 했어.

*   베어메탈 x86에 소프트웨어 개발 첫 шаг

*   FreeDOS 1.4 성공적으로 부트
*   내부 하드웨어 더 살펴보기

### Behringer DDX3216의 기술 세부 사항
*   메인 프로세서: AMD 엘란 SC300 386 SoC (386SX에 통합 UART, PCMCIA, GPIO 등)
*   27C512 64k x 8비트 ROM IC (BIOS용)
*   8 x HYB5117400BJ60 4M x 4비트 RAM (합计 16MB DRAM)
*   1 x UM61256 SRAM (비디오 RAM으로 사용)
*   4 x 29C040-120 플래시 IC (메인 소프트웨어용)
*   내장 LCD (SC300 내장 LCD 인터페이스에 3 x 토шибা T6A39 컬럼 컨트롤러와 1 x T6A40 로우 컨트롤러)
*   토шиб라 TLC16C552 외부 UART (2개의 시리얼 포트와 1개의 병렬 포트)
*   CFカード 연결을 위한 PCMCIA 커넥터 (어댑터 포함)
*   미조립된 인텔 82078 FDC (플로피 디스크 컨트롤러)가 있지만 34핀 커넥터로 연결 안됨

Behringer DDX3216의 하드웨어 구성은 bastante 호환성이 뛰어나서, 일반적인 x86 시스템과 호환될 것으로 보임.

### 베어메탈 x86에 소프트웨어 개발 첫 번째 단계
대부분의 컴퓨터에는 인터넷에서 다운받아 쓸 수 있는 레디메이드 BIOS가 있지만, AMD 엘란 SC300을 위한 BIOS를 찾으려고 하였으나 쉽지 않음. 결국 BIOS를 직접 만들기로 결정.

### DIY x86 바이오스 구현
x86 CPU는 부트 시에 메모리 끝 0xFFF0 위치에서 코드를 가져옴. 이 코드는 인터럽트를 비활성화하고 특정 주소로 점프하는 코드. 이 주소는 ROM의 BIOS 코드가 들어있는 위치.

```ASM
reset_vector:
    nop 
    cli 
    jmp start 
    // padding
    .zero (0x10 - (. - reset_vector) - 8)
    .ascii "06/04/26"
```

이 코드는 인터럽트를 비활성화하고, start 레이블로 점프하는데, 여기서 x86의 "real-mode"로 진입하게 됨. "real-mode"는 8086의 원래 16비트 모드.

reset_vector는 링커 스크립트에 의해 ROM의 0xFFF0 위치에 넣어짐.

### 링커 스크립트
 링커 스크립트는 GCC 컴파일러에 코드를 어떻게 배치할지에 대한 지시사항임. 여기서는 reset_vector를 ROM의 0xFFF0 위치에 넣음.

```plaintext
OUTPUT_FORMAT("elf32-i386")
OUTPUT_ARCH(i386)
ENTRY(reset_vector)
memory {
    ROM (rx) : ORIGIN = 0x0000, LENGTH = 64K
}
SECTIONS {
    .text : { __text_start = .; KEEP((*.text)) (*.text.*) . = ALIGN(2); __text_end = .; } > ROM
    .reset 0xFFF0 : { KEEP((*.reset)) } > ROM
}
```

### 컴파일된 바이너리
```
0000ffa0  00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
0000ffb0  00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
0000ffc0  00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
0000ffd0  00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
0000ffe0  00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
0000fff0  90 fa e9 0b 00 00 00 00 30 36 2f 30 34 2f 32 36
```

이 바이너리를 디스어샘블링 해보면,
```
0000fff0 90              nop
0000fff1 fa              cli
0000fff2 e90b00          jmp    0x0
```

reset_vector는 0x90 (nop命令)로 시작한 다음, 인터럽트를 비활성화하고(start 레이블로) 점프.

## 세그먼트란?
x86에선 세그먼트를 이용해 주소 범위를 넓히는데, 16비트 주소_space만 있으면 64KB만 주소를 지정할 수 있어, 세그먼트를 이용해 1MB까지 주소를 지정할 수 있음.

## 실제 주소는 어떻게 계산되는가?
세그먼트를 이용하여 계산된 주소를 가리켜 "물리 주소"라고 하는데, 이는 (세그먼트 * 16 + 오프셋)으로 계산.

## 메모리 맵
x86 시스템의 메모리 맵은 다음과 같음

```
+--------------------------------------------------+
|                 SYSTEM BIOS                     |
|  64KB ROM                                        |
+--------------------------------------------------+
|  OPTION ROMS OR FREE HIGH MEMORY              |
|  160KB                                            |
+--------------------------------------------------+
|  VIDEO BIOS (VIDEO CARD ROM)                   |
|  32KB ROM                                        |
+--------------------------------------------------+
|  VIDEO RAM (TEXT MODE & VGA GRAPHICS)          |
|  128KB RAM                                       |
+--------------------------------------------------+
|                  CONVENTIONAL MEMORY         |
|  FREE SPACE FOR DOS, PROGRAMS, DRIVERS, ETC.    |
|  ca. 605KB                                       |
+--------------------------------------------------+
```

이 메모리 맵을 보다보니, 왜 DOS와 게임들이 메모리 문제를 겪었는지 이해가 간다. 실제로는 640KB 밖에 되지 않는데도, 그중에서 일부는 다른 용도로 쓰여서, 실제로 쓸 수 있는 메모리는 605KB 밖에 안된다니, 진짜 짜증나는 거 같다.
