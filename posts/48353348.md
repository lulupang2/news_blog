# 10년 전 제온 프로세서로도 충분하다 ㅋㅋㅋㅋ

     2026년 6월 1일, 제 기존 포스트에서는 Gemma 4의 MTP 드래프트를 양자화하고 버리파이에 연결하는 방법을 설명했음.

     이번 포스트에서는 그런 결과를 GPU가 없는 머신에서 돌아가게 하는 방법을 설명할 건데, 개빡치는 점은 많지만 ㅋㅋㅋ.

     제 경우, 래퍼블릭 서버를 하나 구했는데, RAM은 128GB나 되지만 DDR3라서 지금 쓰는 랩탑 램보단 5~6배는 느리더라.

     그리고 CPU는 2016년 제품인 인텔 제온 E5-2620 v4를 쓰고 있는데, 랩탑 CPU보단 5배는慢하다고 보면 되지 않을까?ㅋ

     그리고 GPU는 없고, 그 제온 CPU도 내장 GPU가 없음.

     근데, 그냥 들어봐 ㅋㅋㅋ. olalma를 써도,_gemmalike 모델은 지원하지 않아서 6개월 뒤에야 쓰게 될지도 모를 일임.

     그리고 olalma만큼 좋은 설정값을 노출하지도 않으므로, 그냥 그런대로 돌아가는 수밖에 없음.

     ### 제 시스템 사양

     CPU: 인텔 제온 E5-2620 v4 @ 2.10 GHz

     코어수: 8개, 스레드 수: 16개

     지시어 세트: AVX2 (AVX-512, AVX-VNNI, BF16 지원 안 함)

     캐시: 20MiB L3, 2MiB L2

     메모리: 128GB DDR3

     GPU: 없음

     ## LLM 추론에서 메모리 대역폭이 제일 큰 병목이야.

     토큰을 하나 생성하는데, 몇 기گابايت에 달하는 모델 학습값을 RAM에서 CPU 캐시로 옮겨야 함.

     ChatGPT 쓸 때, 텍스트가 화면에 слово 하나씩 나오는 것을 decoder 단계라고 하는데,

     이 단계는 실제 계산 속도가 병목이라기보단, 메모리 대역폭이 병목이야.

     결국, CPU가 계산을 빠르게 끝내도, 다음 토큰을 계산하기 위해선 메모리에서 모델 값을 읽어와야하고, 그 과정이 너무 느려져.

     이는 CPU든 GPU든, '메모리 월'이라고 부르는 가장 큰 성능 저하 요인 중 하나야.

     ### 제 해결책

     llama-cli에 노출된 거의 모든 최적화 설정값을 적용했어.

     ```
     llama-cli 
     --model gemma-4-26B-A4B-it-Q8_0.gguf 
     --model-draft gemma-4-26B-A4B-it-assistant-GGUF/ 
     wikitext-2-raw_ik-llama-mtp_drafter-conservative/ 
     gemma-4-26B-A4B-it-assistant-Q8_0.gguf 
     --spec-type mtp --draft-max 3 --draft-p-min 0.0 --spec-autotune 
     -cnv --color --jinja --special 
     -sm graph -smgs -sas -mea 256 --split-mode-f32 
     --temp 0.7 -t 8 --parallel 8 
     --cpu-moe --merge-up-gate-experts 
     --flash-attn on --mla-use 3 
     --mlock --run-time-repack --no-kv-offload
     ```
     ## 캐시를 최대로 활용하기

     캐시의 크기와 레이아웃에 데이터가 잘 맞춰져 있어야, 캐시 미스가 발생하지 않고 계산 속도가 빨라질 수 있음.

     ### --run-time-repack

     이는 캐시 레이아웃에 딱 맞는 방식으로 모델 가중치를 재배치하는 설정이야.

     ### --mlock

     이는 모델 가중치를 RAM에 고정시키는 걸 의미해, 시스템이 메모리 부족 상황에서 디스크로 스와핑되는 걸 방지할 수 있음.

     ### --no-kv-offload

     이는 KV 캐시를 GPU로 오프로드하는 것을 막는 설정이야.

     개중에 KV 캐시는 단기 메모리라서, 모델이 계산에 사용하는 가중치를 담고 있는 주기억장치보다 훨씬 빠른GPU에 옮겨서 성능 향상을 기대할 수 있지만, 제 경우에는 GPU가 없으니까 그냥 CPU에서 돌아가게 하고.

     이 캐시를 GPU로 옮기는 작업을 생략해서, 불필요한 대기 시간을 없앨 수 있음.
