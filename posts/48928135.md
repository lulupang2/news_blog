SQLite는 진짜 개 좋은 데이터베이스 엔진인데, 내 개인 프로젝트에서 많이 써왔고, 지역 데이터 저장에 있어서는 업계 표준이라고 해도 되지 않을까? 
일부 서버 소프트웨어에서도 SQLite를 사용하는데, 예를 들어 lobste.rs는 지금 SQLite 기반으로 돌아가는 중임.
전통적인 관계형 데이터베이스 관리 시스템(RDBMS)와 달리, SQLite는 별도의 프로세스가 아니라 라이브러리 형태로 작동하니까 소프트웨어가 자체적으로 독립적이라는 장점이 있음. 
전통적인 파일 포맷과 달리, 커스텀 시리얼라이저와 파서를 작성할 필요도 없어서 일종의 양쪽 세계의 최고를 즐길 수 있는 거임.
근데 여기엔 엄청난 문제가 하나 있 ｰ foreign key 제약조건이 디폴트로 무시됨 ㅋㅋㅋㅋ.
SQLite는 유일하게 디폴트로 외래 키 제약조건을 강制하지 않는 데이터베이스임.
이게 더 臭い건 INTEGER PRIMARY KEY 컬럼이 실제로는 ROWID의 별칭으로 작동하는데, SQLite의 ROWID 할당 알고리즘은ちょっと 복잡한데,
결과적으로는 ID 재사용이 일어나고, 이는 당연히 디버깅을 매우 힘들게 만듦 ㅋㅋㅋㅋ.
예시를 보면,
CREATE TABLE users ( id INTEGER PRIMARY KEY, display_name TEXT );
CREATE TABLE posts ( id INTEGER PRIMARY KEY, user_id INTEGER NOT NULL, content TEXT NOT NULL, FOREIGN KEY(user_id) REFERENCES users(id) );
이러면 user_id 컬럼이 users 테이블의 id 컬럼을 참조하는 외래 키 조건을 설정하는 거임.
디폴트로 외래 키 제약조건이 작동하는 다른 데이터베이스들과 달리, SQLite는 외래 키 제약조건을 무시함 ㅋㅋㅋㅋ.
이거 때문에 DELETE 문을 잘못 사용해서 참조 무결성을 깨트리는 문제가 생길 수 있음 ㅋㅋㅋㅋ.
PRAGMA foreign_keys = ON; 이라고 설정해주면 외래 키 제약조건이 작동하지만, 디폴트로 작동하지 않는다는 점은 매우 문제가 있음 ㅋㅋㅋㅋ.
또 外, SQLite는 컬럼이 잘못된 데이터 타입을 저장할 수 있음 ㅋㅋㅋㅋ.
INTEGER 컬럼에 텍스트를 저장할 수 있는 건 말도 안되는 소리임 ㅋㅋㅋㅋ.
 Strict 테이블을 사용하면 이러한 문제를 해결할 수는데, 모든 테이블에 수동으로 strict 플래그를 추가해야 하니까 매우 불편함 ㅋㅋㅋㅋ.
지금까지 나온 두 문제는 pragma 설정으로 쉽게 해결할 수 있는 문제지만, SQLite에서 기본적으로 이러한 문제가 존재한다는 것은 큰 문제임 ㅋㅋㅋㅋ.
마지막으로 SQLITE_BUSY 오류 문제가 있음 ㅋㅋㅋㅋ.
SQLite는 다중 읽기ですが, 쓰기는 한 번에 하나만 가능함 ㅋㅋㅋㅋ.
디폴트 설정에서는 여러 쓰기 요청이 발생하면 SQLITE_BUSY 오류가 발생하지만, 이거 해결하려면 PRAGMA busy_timeout = 5000; 이런식으로 설정해줘야함 ㅋㅋㅋㅋ.
이 설정으로 인해 락을 획득하기 위해 최대 5초까지 기다리도록 설정할 수 있지만, 디폴트 설정이 이러หา다는 것이 믿기 어렵다는 거임 ㅋㅋㅋㅋ.
요즘에야 이 설정을 알았는데, 이거 모르면 진짜システム에서 버그가 계속 생길 수 있음 ㅋㅋㅋㅋ.
이러한 문제들이 있지만, SQLite는 정말 좋은 데이터베이스임 ㅋㅋㅋㅋ.
단지 디폴트 설정만 조금 더 개선된다면, 훨씬 더 좋은 데이터베이스가 되지 않을까? ㅋㅋㅋㅋ.
