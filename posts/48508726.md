폰트 parsers는 불신할 수 있는 출처로부터 데이터를 처리함으로써, 트루타입 힌트 인터프리터는 보안상 취약한 공격 표면이 되는 거임. FALL 2025 릴리즈에서 우리는 C에서 Swift로 힌트 인터프리터를 다시 작성해서, 메모리 안전성과 성능을 cải善했음. Swift 인터프리터는 평균적으로 13% 빠르뿐만 아니라, 메모리 안전성도 verbess했음.
우리는 Swift 트루타입 힌트 인터프리터의 소스 코드도公開했음. 우리는 Swift에서 유사한 작업을하는 다른 사람들에게 도움이 되기를 바라는데,Swift로의 경험을 공유할 수 있기를希望하기 때문임.
## 트루타입과 힌트 엔진
애플은 1980년대 후반에 트루타ｲ프를개발해서, 1991년에システム 7과 함께 출시했음. 트루타ｲ프는당시 주요한 획득으로, 폰트 개발자들에게 글리프를 표시하는 데 대한 огром한 컨트롤을 제공했으며, 첨단 그리드 피팅 알고리즘과 힌트 엔진이 포함된 특별한 목적의 바이트 코드 인터프리터를 사용했음.
트루타ｲ프는당시의 훨씬 더 약한 컴퓨터에서 모든 것을 수행했기 때문에, 성능에 대해 매우 잘 튜닝되어야했음. 그런 다음 인테넷이 폰트를 사용하는 방식을 혁신했음. 1994년에 트루타ｲ프는 PDF 파일에埋め込을 수 있었으며, 2008년에는 웹 페이지에 삽입할 수 있었고, 여전히 관련성이 있음. 하지만 이러한 새로운 용례는 추가적인 위험을带來했음. 트루타ｲ프는現在 인터넷 어디에서나 불신할 수 있는 폰트에 노출될 수있기 때문임.
## Swift로 재작성
새롭게 작성하는 것은 메모리 안전성을 갖춘 언어로 작성해야 하고, существующ kodbase에서 동등한 성능 수준을 제공할 수 있어야했음. Swift는 당연히 이러한 과제에 적합한選択였음. 이 프로젝트의 성공을위한 바이너리 호환성은 매우 중요했음. 기존 프로그램들은 새로운 구현이 있는지 모르고, 동일하게 작동해야했음. 이것은 단순히 인터페이스 호환성뿐만 아니라, 픽셀-일치한 글리프 렌더링도 의미했음. 힌트는 글리프의 화면상의 모양을 극적으로 변경할 수 있으므로, 인터프리터의 동작에 작은 변화도巨大한 사용자可視的 변화로 이어질 수있음.
우리는 이 프로젝트에서 정당성을 정의하기 위해 C 구현의 출력과 완전히 호환되는 것을 의미했음. 
## 정당성 검사
우리는 정당성을 보장하기 위해 두 가지 테스트 스위를 개발했음. 첫 번째는 유닛 테스트 스위트로, 두 가지 구현 모두를 대상으로 할 수 있었으며,Swift 인터프리터에 대한 99.7%의 코드 커버리지를 제공했음. 이 스위트는 오픈 소스 릴리즈와 함께 제공된 Swift 인터프리터에 포함되어있음.
다음으로 실제 작업負荷를 나타내기 위해, 우리는 1,000만 개의 PDF 파일에서 구성된 코퍼스를 4,200 개로 축소했으며 코드 커버리지를 손실하지 않았음. 축소된 코퍼스의 문서에는 25,572개의 폰트와 총 2,700만 개의 글리프가 포함되어 있었으며, 각 글리프를 4가지 다른 변환을 적용해서 렌더링했으며, 생성된 비트맵을 참조 인터프리터와 비교했음. 이것은 새로운 인터프리터의 호환성에 대한 확신을줬음.
프로젝트가 끝날 무렵, 우리는 Swift 인터프리เตอร 자체에 쓴 줄 수의 거의 4배에 해당하는 양의 테스트 코드를 작성했음. 
## 성능 개선
새로운 구현이 모든 테스트를 통과한 후, 우리는 성능에 집중했음.PDF 렌더링 시간을 사용한 성능을 높은 수준에서 평가했으며, 세 가지 다른 폰트의 모든 글리프를 렌더링하는 벤치마크를 사용하여 개선 사항을 반복했음. 이러한 개선은 네 가지 주요 카테고리로 분류할 수있음.
### 실행 오버헤드 최소화
Swift는 공유 참조 유형의 수명 주기를 관리하기 위해 자동 참조 카운팅을 사용하고, 데이터 구조에 중복 접근을 방지하기 위해 런타임 배타적 확인을 사용했음. 이러한 오버헤드의来源은 souvent에 aliasing으로 인해 악화되는데, 이는 인터프리터 사양에서 설계적이지 않은 양의 aliasing이 포함되어 있기 때문임. 이러한 오버헤드의 출처는 복사 가능성을 포기하고, 구조체 대신 클래스를 사용하여 suốt아키텍처에 '~Copyable' 값 유형을採用함으로써 제거할 수있음. Swift 6.2에서 도입되어 macOS 10.14.4 및 iOS 12.2로의 역 방향 지원을 포함한 스팬(Span)은 이러한 유형의 시퀀스를 효율적으로 작동할 수있게 해줬음. 
### 데이터 이동 최소화
때때로 언어 경계를 넘나들 때 구조화된 데이터의 '형태'를更改하여 반대편의 관용구와 더 잘 맞게 만들고 싶을 때가 있음. Swift에서 글리프 아웃라인은 각 점마다 플래그를 갖는 시퀀스로 표현되는데,curve上에 있지, 각 축에서 접촉되었는지에 대한 플래그가 있으며, 세 개의 좌표 쌍(coordinate Pairs)이 있음. C코드에서는 이러한 점을 여덟 개의 배열을 가진 하나의 구조체에 저장했음. 이것은 캐시 친화적이기 때문에 성능면에서 좋음. 하지만 Swift에서 데이터를 포인트 요소의 컬렉션으로公開하면 소스 코드가 더 읽기 쉬워짐. 
초기 크로스 언어 브리징 코드는 속도, 安全性, 간단성을 우선적으로 고려해서 C 구조체에서 글리프 데이터를 Swift로 복사하고 완료되면 다시 뒤로 복사했던 것으로, 처음에는 이러한 복사 작업이 새로운 인터프리터의 실행 시간의 20%를 차지했음. 하지만 결국 우리는 프로젝션 유형을 사용해서 기본적인 C 구조체에 대한 안전한 접근을 제공하게되었습니다. 
Swift는 기본적인 데이터 구조체를 변형시키거나 복사하지 않고도, 읽기 쉬운 소스 코드를 제공할 수있게 해줬음. 
### 단기 할당 제거
런타임 오버헤드는 총과적으로 비용을 부과하며, 언어 경계를 넘나들 때 데이터를 복사하는 것이 특정한 핫スポ트로 나타났으며, 단기 메모리 할당의 비용은 여러 가지 방식으로 나타날 수있음. 필터링이나 맵핑과 같은 작업은 메모리를 할당하거나 필요한 경우에만 할당해야함. Swift 표준 라이브러리는 '.lazy.map'나 '.lazy.filter'와 같은 기능을 제공하지만, 모든 경우에 작동하지는 않음. 특정 논리만 반복문으로 반복하는 경우에는 루프를 사용하고, 필요한 경우에만 로컬 변수를 사용할 수있음.
함수가 호출자에게 결과를 반환할 때 할당된 메모리를 전달할 필요가없음. 예를 들어, 인터프리터의 스택에서 마지막 n 요소를 꺼내야하는 경우, 明顯한 구현 방법은 다음과 같으며 메모리를 할당해서 반환해야함.
```swift
mutating func pop(count n: Int) -> [Element] {
    defer { items.removeLast(n) }
    return Array(items.suffix(n))
}
```
이 작업을 안전성을 손상시키지 않고 최적화하는 것은 호출자가 블록을 전달하여 스택 요소에 접근할 수 있는 지속성 전달 방식으로 이어짐. Swift의 컴파일 타임 배타적 확인은 스택이 블록 내에서 수정되지 않을 것임을 보장했으며, 인터페이스는 힙 할당이나 요소 복사와 같은 필요성을 구조적으로 제거하기도했음.
```swift
mutating func pop<R, E: Error>(count n: Int, _ op: (borrowing Span<Element>) throws (E) -> R) throws (E) -> R {
    defer { items.removeLast(n) }
    return try op(items.span.extracting(last: n))
}
### 동적 디스패치
추상화 메커니즘으로 프로토콜, 제네릭스, 상속 등이 있는데, 런타임에 메소드호출 간접 참조가 나타날 수 있음. 이러한 오버헤드는 часто 최적화에 의해 제거될 수 있지만, 모든 조건에서 불가능한 것이 아님. 우리의 경우, 추상화를過度하게 일반화하지 않고, 툴체인이 인라이닝할 수 있도록 유도하면, 바운드 검사와 모든 제네릭 컨텍스트를 스페셜화하기 위한 최적화를가능하게 할 수있었음. 프로파일링 중에熱 경로에서 unspecilaized 제네릭스나 프로토콜 증거 테이블을 보게 되면, 최적화기가 충분한 비빌리티를가지지 못해서호출 사이트를최적화할 수없는것이라는 신호이므로, 코드를 더 많이 인라이닝하고, 추상화 수준을 낮추는 것이必要할 수 있음.
