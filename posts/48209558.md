###_SBCL: 더 간결한 어셈블리 코드_

딥러닝이 발전하고, 프로그래밍 언어도 점점 발전하는데, 여전히 최적화된 코드에 대한需求은 계속 존재함. 이번에는 10개의 슬롯만 있는 스택 기반 가상머신인 SBCL에서 어떻게 어셈블리 코드를 최적화할 수 있는지에 대한 글임.

#### NEXT 시퀀스
NEXT 시퀀스는 기존에 base 레지스터 отсутств했을 경우 effective 주소를 인덱싱 하는데 사용되었음. 하지만 이것은 효율적인 인코딩이 아님. 기계어 코드의 차이를 보자:
* 이전(14 바이트): 
  - `MOV EAX, [RDI]`
  - `ADD RDI, 4`
  - `ADD RAX, RSI`
  - `JMP RAX`
* 이후(9 바이트): 
  - `MOV EAX, [RDI]`
  - `ADD RDI, 4`
  - `ADD RAX, RSI`
  - `JMP RAX`
 NEXT의 정의를 수정했지만, 아직도 구형 기계어 코드를 사용하고 있다고 함.

#### F18
F18을 다시 살펴봄. Chuck Moore의 작품 답게, 정상인지 광ki인지 구분이 안됨 ㅋㅋ. 한 가지 주목할 점은 스택의 크기가 매우 작다는 것임. 10개의 슬롯 밖에 없으며, 오버플로우/언더플로우 트랩도 없음. 이유는_overflow가 발생하면 잘못된 코드라는 뜻이며, silent overflow는 일부로 발생시키는 거임.

#### x87
x87도 비슷한 경우임. djb의 글에서 x87의 회전 스택을 잘못 사용하고 있다고 말했었음. 하지만, FXCH를 잘 활용하면 회전 스택은 레지스터와 같거나 더 나을 수 있다고 함. 예를 들어, 다음 루프는 데이터 이동 없이 цик을 실행할 수 있음:
```assembly
LOOP:
  FADD ST(1), ST(0)
  FXCH ST(1)
  JMP LOOP
```
#### 스택 기반 가상머신
이제 8개의 슬롯만 있는 스택 기반 가상머신에 대해 이야기 해 봄. 가장 이상적인 경우는 모든 것을 레지스터에 저장하는 것임. 하지만, push와 pop이 복잡해지기 때문에 Forth 엔진은通常 스택의 상위 1~2개의 요소만 캐싱함.

#### 스택 포인터
스택 포인터가 8개의 값을 가질 수 있기 때문에, primitive를 각 값에 대해 특화시킬 수 있음. primitive는 다음으로 넘어가는 코드를 가리키는 포인터를 포함하고 있음.
```assembly
MOV EAX, [RSI]
ADD RSI, 4
ADD RAX, RDI
JMP RAX
```
#### emit-code
emit-code 함수는 primitive 코드를 생성하여 각 슬롯에 맞게 padding을 하여 코드를 맞춤. primitive 함수를 호출하여 코드를 생성할 수 있음.
```lisp
(emit-all-code 'swap 'dup 'drop 'add 'sub)
```
#### 결과
위 함수들을 호출하면 각 primitive에 대한 코드가 생성됨. 다음은 swap 함수에 대한 코드임:
```assembly
XCHG R8D, R9D
MOV EAX, [RDI]
ADD RDI, 4
ADD RAX, RSI
JMP RAX
```
이상임. 개빡치는 점은, 이렇게 최적화된 코드도 결국에 인간이 읽기 어렵다는 거임 ㅋㅋㅋㅋ. 근데, 어셈블리 언어를 공부하면 컴퓨터의 동작 원리를 더 잘 이해할 수 있고, 더 효율적인 프로그램을 개발할 수 있음.，所以, 어셈블리 언어를 공부해 보기로 함 ㅋㅋ. 끝!!
