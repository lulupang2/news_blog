# 오픈코드 쓰지 마세요 ㅋㅋㅋ
   오픈코드가 뭔지 모르면, 타입스크립트로ทำ된靴이 인간얼굴을 계속 밟고있는거 생각해봐啦 ㅋㅋㅋㅋ 그靴은 보안과 시스템 소프트웨어에 관해서 전자計算機가 발명된 1940년대부터 지금까지 배운 모든것을 무시하는거임 ㅋㅋㅋㅋ 
   제작자들은 오픈코드를 AI 코딩 에이전트라고 설명했지만, 나의 결론은 오픈코드는 그냥 쓰레기라고 생각함 ㅋㅋㅋㅋ 
   이 포스트는 두 부분으로 나뉘어있음 ㅋㅋㅋㅋ 하나는 짜증나는 부분이고, 하나는 위험한 부분임 ㅋㅋㅋㅋ 두번째部分이 더 길음 ㅋㅋㅋㅋ 
   나는 오픈코드 깃허브 버전 baef5cd4의 소스코드를 참조해서 이 포스트를 작성했음 ㅋㅋㅋㅋ 여기서 설명한 내용은 보안 관련 내용이 아님을 밝힘 ㅋㅋㅋㅋ 
   오픈코드는 기본적으로 llm | bash를 위한 웹스택 툴이고, 나는 pipe부분에서 발생하는 문제를 설명할거임 ㅋㅋㅋㅋ 
   이 툴이 어떻게 실패하는지 설명하기 전에, LLM를 사용하는 것과 사용하면 máy가 쉽게 공격당하거나 삭제되는 것 사이의 구분선을 그어놨어 ㅋㅋㅋㅋ 
   추가로, 로컬 LLM에 대한 생각을 간단하게 덧붙였음 ㅋㅋㅋㅋ 
   ## 짜증나는 점들 ㅋㅋㅋㅋ
   보안을 잠시 제쳐두고, 오픈코드가 툴로서 어떻게 실패하는지 살펴보겠음 ㅋㅋㅋㅋ 
   베데스다 효과처럼 오픈코드에서 무엇이 버그이고, 무엇이 의도적인 디자인인지 구분하기가 어려워서, 그냥 '짜증나는'이라고 설명하기로 했음 ㅋㅋㅋㅋ 
   ### 프롬프트 캐시 미스 ㅋㅋㅋㅋ
   대부분의 로컬 LLM서버는 오픈AI의 /v1/chat/completions API를 사용함 ㅋㅋㅋㅋ 아이디어는 간단함 ㅋㅋㅋㅋ 
   당신은 POST요청을 날려서 전체 대화 내용을 보내면, SSE이벤트를 받아서 응답을 받을 수 있음 ㅋㅋㅋㅋ 
   하지만 업로드 비용은 제곱으로 증가하고, 다운로드는 JSON에 반복되는 메타데이터로 감싸져있어서 엄청 느림 ㅋㅋㅋㅋ 
   툴을 호출하는 것에는 "double JSON encoding"이라는게 있어서, JSON으로 인코딩된 델타를 여러개 만들어서, 다른 JSON으로 디코딩할 수 있음 ㅋㅋㅋㅋ 
   이 세팅에는 상태가 없다는 하나의 장점이 있음 ㅋㅋㅋㅋ 보통 상태가 없다는 것은 상태를 많이 사용해서 빠르게 만든다는 뜻임 ㅋㅋㅋㅋ 
   서버는 평가 결과를 캐시함 ㅋㅋㅋㅋ请求을 받으면, 캐시된 가장 긴 프리픽스를 찾고, 프리픽스 끝에서 마지막으로 보낸 메시지까지 평가함 ㅋㅋㅋㅋ 
   새로운 토큰을 생성해서 끝을 만날 때까지 계속 생성함 ㅋㅋㅋㅋ 
   나는 Qwen3.6-27B dense를 M4 맥스에서 사용했음 ㅋㅋㅋㅋ 메모리 밴드위드는 khá 좋지만, 토큰 생성은_PREFILL에서非常计算 密集적으로 작동함 ㅋㅋㅋㅋ 
   만약 서버가_prompt cache를 찾을 수 없으면, 최대 GPU 사용으로 10분정도 걸릴 수 있음 ㅋㅋㅋㅋ 
   그건 괜찮음 ㅋㅋㅋㅋ 하지만 오픈코드는 이런 점을 잘 모르겠음 ㅋㅋㅋㅋ 
   * 파일시스템을.glob해서 AGENTS.md를 읽고, 다시 읽어서 캐시를 초기화함 ㅋㅋㅋㅋ
   * 에이전트에서 사용자로 전환할 때, 도구 호출 결과를_pruning해서 캐시 무효화함 ㅋㅋㅋㅋ 
   * 도구 호출 결과는固定距离(const PRUNE_PROTECT = 40_000)만큼 잘라서 캐시에서 제거함 ㅋㅋㅋㅋ 
   * 에이전트에서 사용자로 전환할 때, prompt cache를_trash하기 때문에, 캐시를 새로 생성해야 함 ㅋㅋㅋㅋ 
   * 개인적인 즐겨찾기: 현재 날짜를_turn-0 system prompt에 넣어서 캐시 무효화함 ㅋㅋㅋㅋ 
   이 캐시 무효화는 prompt cache misses로 이어지는데, 이는 여기에 설명한 것만이 아님 ㅋㅋㅋㅋ 
   ### 프루닝 ㅋㅋㅋㅋ 
   이전 섹션에서 설명한 프루닝은_prompt cache misses로 이어지는데, 이는 여기에 설명한 것만이 아님 ㅋㅋㅋㅋ 
   나는 캐시 무효화를 비활성화함 ㅋㅋㅋㅋ 다른 큰 문제는 초기 읽기 보호가 없음 ㅋㅋㅋㅋ 
   오픈코드에서 이것은非常broken임 ㅋㅋㅋㅋ 
   * 예를 들어, 새로운 세션을 시작하고, 에이전트에게 스펙이나 구현 계획을 읽으라고 하면, 캐시가_pruning으로 인해 삭제됨 ㅋㅋㅋㅋ 
   * 에이전트는 코드를 작성하지만,_spec을 참조할 수 없음 ㅋㅋㅋㅋ 
   * 도구 결과는_pruning으로 인해 캐시에서 제거됨 ㅋㅋㅋㅋ 
   * 에이전트는 스펙을 参考할 수 없으므로, 코드를 작성할 수 없음 ㅋㅋㅋㅋ 
   ### 컴팩션 ㅋㅋㅋㅋ 
   * 세션을_summary하고 싶으면 어떻게 해야 할까? _prefill해서 새_prompt로 만들기? ㅋㅋㅋㅋ 
   * 나는 Explicit handoff를 사용해, 에이전트에게 노트를 작성하게 함 ㅋㅋㅋㅋ 
   * 컴팩션은_leaky abstraction으로, 유한한 컨텍스트 윈도우를 무한한 윈도우로 보이게 만듦 ㅋㅋㅋㅋ 
   * 나는 컨텍스트 윈도우와_prompt cache를_first-class feature로 만들어서, primitives를 노출시키는 것이 좋다고 생각함 ㅋㅋㅋㅋ
