### 요약
리버스 프록시에서 HTTP 말고 패스트 CGI를 사용하는게 더 좋다고说的 거임. 패스트 CGI가 30년 전 나왔지만 여전히 melhores라고 함.

### 패스트 CGI란?
패스트 CGI는 리버스 프록시에서 백엔드와 통신하기 위한 프로토콜임. HTTP와 다르게, 패스트 CGI는 더 간단하고 안정적인 프로토콜이라고 함.

### 문제는 HTTP
HTTP는 리버스 프록시에서 사용하기에는 문제가 많다고 함. HTTP는 너무 복잡하고, 구현하기도 힘들고, 보안에도 문제가 많다고 함. 특히, HTTP desync 공격이나 request smuggling과 같은 보안 문제가 많다고 함.

### 패스트 CGI의 장점
패스트 CGI는 HTTP보다 더 간단하고 안정적인 프로토콜이라고 함. 패스트 CGI는 HTTP와 다르게, 메시지의 경계를 명확하게 정의하기 때문에 desync 공격이나 request smuggling과 같은 보안 문제가 없다함. 또한, 패스트 CGI는 클라이언트와 백엔드 사이의 중계자 역할을 하는 리버스 프록시가 클라이언트의 IP 주소를 백엔드에 전달할 수 있는 방법을 제공한다함.

### 예제 코드
Go 언어에서 패스트 CGI를 사용하는 예제 코드如下:
```go
l, _ := net.Listen("tcp", "127.0.0.1:8080")
fcgi.Serve(l, handler)
```
이 코드는 TCP 소켓을 열고, 패스트 CGI 프로토콜로 백엔드와 통신하기 위한 핸들러를 설정하는 코드임.

### 결론
리버스 프록시에서 패스트 CGI를 사용하는게 더 좋다고 함. 패스트 CGI는 HTTP보다 더 간단하고 안정적인 프로토콜이고, 보안에도 더 좋다고 함. 30년 전 나온 패스트 CGI 프로토콜이여도 여전히 meilleurs라고 함. ㄹㅇ 개빡치는 점
