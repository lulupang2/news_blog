# wasm 게임 엔진의 충격적인 용량
와!_fsm 게임 엔진이 도커 이미지보다 가벼워서 개놀램 ㅋㅋㅋ;
게임 스킬론을 웹어셈블리로 내보냈는데 용량이 너무 작아서 놀랐어. 3D 엔진, GL 호환 렌더러, Jolt 물리학, GDScript 런타임, Ink 내러티브 인터프리터 다 포함한 게임 엔진이 35MB로 돌아감;;;;
그리고 브라우저에서 아무 설치 없이 실행 가능 ㅋ

# 크기 비교
* 구글 홈페이지: 10MB (43개의 요청)
* 이 게임 (Godot 4, 풀 엔진): 35MB
* 페이스북 홈페이지: 44MB (379개의 요청)
* 라이브킷/라이브킷-서버 (Go, WebRTC): 75MB
* 파이썬 도커 이미지 (slim-trixie): 144MB
* 파이썬 도커 이미지 (slim-trixie) + 최소 종속성: 282MB
* REST API (직장에서 사용): 300-400MB
* 노드 도커 이미지 (latest): 421MB
* ghcr.io/gohugoio/hugo: 423MB
* 파이썬 기반 AI 에이전트 (직장에서 사용): 1.45GB

# 왜 wasm이 널리 사용되지 않는가?
와! Go는 하나의 해결책이 될 수 있지만, 아직은预览임;;; сокет 및 스레드가 없는 표준 런타임이 문제임. Zig는 가까이 있지만 아직은 아니야.目前 praktische 선택지는 Rust와 C/C++ 밖에 없음.
그렇지만!_cloudflare 워커는 wasm 모듈을 로드할 수 있고, containerd에는 runwasi가 있고, Kubernetes에는 kwasm 实验이 있으며, WASI 런타임도 존재해!
왜 wasm의 채택이 정체되었을까? 전송 크기는 이미 10배로 충분히 좋은데, 왜 표준으로 사용되지 않는 걸까?_ARM 노드가 몇 년 전에Wide로 사용되지 않았던 것 처럼;; 값이 rẻ고, 밀도가 높고, 광범위하게 사용 가능했지만, 여전히 기본 선택이 아니었음
