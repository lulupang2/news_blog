# Conventional Commits는 쓰레기야 
 Conventional Commits를 본 적이 있을거야. 오픈소스 프로젝트에서 쓰는 커밋 메시지 포맷이거나, 직접 기여한 프로젝트에서 강제되는 커밋 포맷일수도 있음. 하지만 나에게 Conventional Commits는 그냥 쓰레기야.

 ## Scope > Type 
 커밋 메시지에서 scope는 가장 중요한 부분이야. 하지만 Conventional Commits에서는 scope를 옵션으로 두고, type을 우선시함. 이건完全 반대야.

 커밋 메시지의 형식은 다음과 같음: `<type>[optional scope]: <description> [optional body] [optional footer(s)]`. 하지만 이 형식은 type을 우선시하고, scope를 옵션으로 두고 있어. Scope가 중요하지, type이 중요한게 아님.

 ## Type은 쓸모없고 제한적이야 
 Type은 커밋 메시지에서 불필요한 요소야. 커밋 메시지의 설명에서 이미 type을 유추할 수 있기 때문이야. 예를 들어, `fix(compiler): prevent namespaced SVG <style> elements from being stripped`라는 커밋 메시지는 이미 fix라는 TYPE을 가지고 있지. 

 ## Conventional Commits는 약속을 지키지 않음 
 Conventional Commits는 자동으로 CHANGELOG를 생성하고, 의미 있는 버전 번호를 결정하며, 팀원들과의 소통을 개선한다고 하지만, 이 모든 약속을 지키지 못하고 있어. 

 ## 자동으로 CHANGELOG를 생성한다는것은 쓸모없음 
 CHANGELOG는 사용자에게 새로운 기능을 설명하기위한 문서이지, 커밋 메시지의 기록이 아님. 

 ## 의미 있는 버전 번호를 결정한다는것은 틀림 
 버전 번호는 사람이 직접 결정해야 함. 자동으로 결정하는 것은 많은 경우에 정확하지 않음. 

 ## 팀원들과의 소통을 개선한다는것은 개뿔 
 Conventional Commits는 팀원들이 커밋 메시지를 더 잘 이해하도록 도와주지 않음. 

 ## 건전한 대안 
 successful한 프로젝트들은 scope-prefixed commit messages를 사용하고 있음. 예를 들어, 리눅스 커널은 subsystem을 scope로 사용하고, Go 프로젝트는 package path를 scope로 사용하고 있음. 

 ## 결론 
 Conventional Commits는 쓰레기야. Scope-prefixed commit messages를 사용하도록 하자.  ㅋㅋㅋ
